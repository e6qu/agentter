# How the Azure Terraform Providers Are Built

This note covers the three Terraform providers in the Azure ecosystem and, crucially,
the code-generation pipeline that sits underneath the mainstream one. The headline:

- **`hashicorp/terraform-provider-azurerm`** — the mainstream provider, **mostly hand-written Go**,
  but every resource calls into a **generated SDK**, `hashicorp/go-azure-sdk`.
- **`hashicorp/go-azure-sdk`** is **generated by `hashicorp/pandora`** from Microsoft's Swagger in
  `Azure/azure-rest-api-specs`. This is HashiCorp generating its **own** Azure SDK rather than using
  Microsoft's `azure-sdk-for-go`.
- **`Azure/terraform-provider-azapi`** — a **generic, schema-driven** provider that talks to ARM REST
  APIs directly using **embedded Azure type definitions** copied from the **Bicep types** project,
  which is itself generated from `azure-rest-api-specs`.

The single most important structural fact: **pandora consumes the *same* `Azure/azure-rest-api-specs`
Swagger** that AutoRest/TypeSpec consume for the official `azure-sdk-for-*` SDKs and that the Bicep
types (and therefore azapi) consume. Three different generators, one source of truth.

## Layer 1 — `terraform-provider-azurerm`: hand-written resources on a generated SDK

### Hand-authored resources

`azurerm` resources are **hand-written Go**. Each service lives under
`internal/services/<service>/` with one file per resource/data-source, e.g.
`internal/services/appconfiguration/app_configuration_resource.go`. These are authored by hand;
there is a per-service `registration.go`, hand-written acceptance tests, schema definitions, and
expand/flatten code.

The provider currently runs on **both** plugin frameworks at once and muxes them. `go.mod` pins:

```
github.com/hashicorp/terraform-plugin-sdk/v2 v2.38.1          # untyped, legacy
github.com/hashicorp/terraform-plugin-framework v1.16.1       # new framework
github.com/hashicorp/terraform-plugin-mux v0.21.0             # serves both
github.com/hashicorp/go-azure-sdk/resource-manager v0.20260520...
github.com/hashicorp/go-azure-sdk/sdk v0.20260520...
github.com/hashicorp/go-azure-helpers v0.80.0
```
([`go.mod` @ 0599a99](https://github.com/hashicorp/terraform-provider-azurerm/blob/0599a9911725e7a2aa802932cba5c539f36233c1/go.mod))

The high-level architecture doc states the codebase "uses a mixture of both (primarily the Untyped
Data Sources/Resources) — in time we plan to migrate across to using Typed Data Sources/Resources
instead", and that this typed abstraction "will allow us to switch from using the Terraform Plugin
SDK to Terraform Plugin Framework"
([`contributing/topics/high-level-overview.md` @ 0599a99](https://github.com/hashicorp/terraform-provider-azurerm/blob/0599a9911725e7a2aa802932cba5c539f36233c1/contributing/topics/high-level-overview.md)).
That doc also names the dependency explicitly: "**go-azure-sdk** — an opinionated Go SDK generated
by HashiCorp for interaction with Azure Resource Manager."

The typed abstraction layer lives in `internal/sdk/` (e.g. `resource.go`, `framework_resource.go`,
`framework_migration_experimental.md`), which wraps both the Plugin SDK v2 and the Plugin Framework
behind a single `sdk.ResourceMetaData`-style API
([`internal/sdk/` @ 0599a99](https://github.com/hashicorp/terraform-provider-azurerm/tree/0599a9911725e7a2aa802932cba5c539f36233c1/internal/sdk)).
Typed resources use it, e.g. `internal/services/nginx/nginx_api_key_resource.go` with
`func(ctx context.Context, metadata sdk.ResourceMetaData) error`.

### …calling into the generated SDK

The load-bearing point: every resource, typed or untyped, calls into `go-azure-sdk`. Concrete
example — the hand-written `app_configuration_resource.go` imports generated service packages:

```go
import (
    "github.com/hashicorp/go-azure-sdk/resource-manager/appconfiguration/2024-05-01/configurationstores"
    "github.com/hashicorp/go-azure-sdk/resource-manager/appconfiguration/2024-05-01/deletedconfigurationstores"
    "github.com/hashicorp/go-azure-sdk/resource-manager/appconfiguration/2024-05-01/operations"
    "github.com/hashicorp/go-azure-sdk/resource-manager/appconfiguration/2024-05-01/replicas"
    "github.com/hashicorp/go-azure-sdk/sdk/client"
    "github.com/hashicorp/go-azure-sdk/sdk/client/pollers"
    "github.com/hashicorp/terraform-provider-azurerm/internal/sdk"
    ...
)

func resourceAppConfiguration() *pluginsdk.Resource { ... }   // still Plugin SDK v2 here
```
([`app_configuration_resource.go` @ 0599a99](https://github.com/hashicorp/terraform-provider-azurerm/blob/0599a9911725e7a2aa802932cba5c539f36233c1/internal/services/appconfiguration/app_configuration_resource.go))

So: **hand-written resource logic + schema, on top of a generated API client**. The provider also
embeds a few other SDKs for services pandora doesn't cover yet (the overview lists `Azure SDK for
Go`, `giovanni` for Storage, `hamilton` for Graph), but the dominant path is go-azure-sdk.

## Layer 2 — `go-azure-sdk` + `pandora`: HashiCorp generating its own Azure SDK

`go-azure-sdk` is **generated**, not hand-written. Its root README says plainly: "The SDKs within
this repository are generated using the data in [the Azure Rest API Specifications
repository](https://github.com/Azure/azure-rest-api-specs)"
([`README.md` @ 14aa3d2](https://github.com/hashicorp/go-azure-sdk/blob/14aa3d220c9eefb0606d2772b42942e62901d622/README.md)).
The generated tree is structured exactly as azurerm imports it:
`resource-manager/<service>/<api-version>/<resource>/`, e.g.
`resource-manager/appconfiguration/2024-05-01/configurationstores/` containing `client.go`,
`version.go`, `method_*.go`, `model_*.go`, `id_*.go`, `constants.go`, `predicates.go`, `README.md`
([dir @ 14aa3d2](https://github.com/hashicorp/go-azure-sdk/tree/14aa3d220c9eefb0606d2772b42942e62901d622/resource-manager/appconfiguration/2024-05-01/configurationstores)).

### Concrete example — a generated file header

The generated `version.go` carries the Microsoft copyright header (because its source is the
Microsoft spec — more on that below) and bakes in the API version:

```go
package configurationstores

// Copyright (c) Microsoft Corporation. All rights reserved.
// Licensed under the MIT License. See NOTICE.txt in the project root for license information.

const defaultApiVersion = "2024-05-01"

func userAgent() string {
    return "hashicorp/go-azure-sdk/configurationstores/2024-05-01"
}
```
([`version.go` @ 14aa3d2](https://github.com/hashicorp/go-azure-sdk/blob/14aa3d220c9eefb0606d2772b42942e62901d622/resource-manager/appconfiguration/2024-05-01/configurationstores/version.go))

The model files share the same header
([`model_configurationstore.go` @ 14aa3d2](https://github.com/hashicorp/go-azure-sdk/blob/14aa3d220c9eefb0606d2772b42942e62901d622/resource-manager/appconfiguration/2024-05-01/configurationstores/model_configurationstore.go)).
The repo `NOTICE.txt` attributes the derivation: "This codebase contains a modified portion of code
from Microsoft which can be obtained at … `Azure/azure-rest-api-specs`"
([`NOTICE.txt` @ 14aa3d2](https://github.com/hashicorp/go-azure-sdk/blob/14aa3d220c9eefb0606d2772b42942e62901d622/NOTICE.txt)).

### The generator: pandora

`hashicorp/pandora` is the toolchain that produces go-azure-sdk. Its `tools/` directory hosts the
pipeline ([dir @ dca5fb0](https://github.com/hashicorp/pandora/tree/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools)):

1. **`importer-rest-api-specs`** — parses the Azure Swagger into pandora's intermediate **data model**
   ("API Definitions").
2. **`data-api`** — serves those definitions over HTTP for downstream generators.
3. **`generator-go-sdk`** — reads the Data API and emits `go-azure-sdk`.
4. **`generator-terraform`** — reads the Data API and emits Terraform resources for azurerm.

The pandora README describes this as an automated chain: when a PR updates the imported specs, the
importer runs, then the Go SDK generator, then the Terraform generator, each opening a PR
([`README.md` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/README.md)).

**Importer (Swagger → data model).** `importer-rest-api-specs` depends on the `go-openapi` Swagger
libraries — `go-openapi/loads`, `go-openapi/spec`, `go-openapi/analysis`
([`go.mod` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/importer-rest-api-specs/go.mod)).
Its `architecture.md` lays out the stages: parse Swagger tags → parse resource IDs → parse
operations → apply data workarounds → output to the Data API
([`architecture.md` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/importer-rest-api-specs/architecture.md)).
Critically, the Azure spec repo is wired in as a **git submodule**:

```
[submodule "submodules/rest-api-specs"]
    path = submodules/rest-api-specs
    url = git@github.com:Azure/azure-rest-api-specs.git
```
([`.gitmodules` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/.gitmodules)).
The resulting intermediate model is checked into `api-definitions/` (with a
`handwritten-resource-manager/` sibling for hand-authored overrides/patches)
([`api-definitions/` @ dca5fb0](https://github.com/hashicorp/pandora/tree/dca5fb0f724ef92619919810f0e0a216a57d2e66/api-definitions)).

**Generator (data model → Go SDK).** `generator-go-sdk` "retrieves a list of Resource Manager
Services from the Data API … and then iterates over each (Generation) Stage to output the files as
needed"
([`README.md` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-go-sdk/README.md)).
Its `internal/generator/` directory maps one-to-one onto the generated files: `stage_clients.go`,
`stage_version.go`, `stage_models.go`, `stage_methods.go`, `stage_ids.go`, `stage_constants.go`,
`stage_readme.go`, with matching `templater_*.go` templaters
([`internal/generator/` @ dca5fb0](https://github.com/hashicorp/pandora/tree/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-go-sdk/internal/generator)).

**Concrete example — the generator emits the exact header seen in go-azure-sdk.** The "Microsoft
Corporation / MIT License" comment is produced by `copyrightLinesForSource`, which keys off where the
data came from:

```go
func copyrightLinesForSource(input models.SourceDataOrigin) (*string, error) {
    if input == models.HandWrittenSourceDataOrigin { ... }          // IBM Corp header
    if input == models.MicrosoftGraphMetaDataSourceDataOrigin { ... }
    if input == models.AzureRestAPISpecsSourceDataOrigin {
        out := `
// Copyright (c) Microsoft Corporation. All rights reserved.
// Licensed under the MIT License. See NOTICE.txt in the project root for license information.
`
        return &out, nil
    }
    ...
}
```
([`internal/generator/helpers.go` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-go-sdk/internal/generator/helpers.go#L77))
This closes the loop end-to-end: the header in go-azure-sdk's `version.go` is literally what
pandora's generator writes when the source-data origin is `AzureRestAPISpecsSourceDataOrigin`.

**Why this matters:** unlike Microsoft's own `azure-sdk-for-go` (AutoRest/TypeSpec-generated),
HashiCorp generates *its own* opinionated Go SDK from the same upstream specs, tailored to what the
Terraform provider needs (consistent resource-ID handling, pollers, `commonids`, etc.). The pandora
README notes API versions are added on a "what do you want to use it for" basis precisely because the
SDK is provider-driven, not a general-purpose mirror.

`generator-terraform` is the same machinery pointed at azurerm: it "generates a Terraform SDK using
the information available in the Data API," updating `internal/clients/`, `internal/provider/`, etc.
for a service ([`README.md` @ dca5fb0](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-terraform/README.md)).
So azurerm is moving from "all hand-written" toward "scaffolding generated by pandora, behaviour
hand-finished" — but today the bulk of resources are still hand-authored. `?? verify` what fraction
of azurerm resources are currently produced via generator-terraform vs hand-written (noted in
DO_NEXT).

## Layer 3 — `terraform-provider-azapi`: generic, schema-driven, embedded ARM types

`azapi` is the opposite design point. Its README calls it "a very thin layer on top of the Azure ARM
REST APIs" whose purpose is to "complement the AzureRM provider by enabling the management of Azure
resources that are not yet or may never be supported in the AzureRM provider such as private/public
preview services and features"
([`README.md` @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/README.md)).
There are **no per-resource Go files**: a single generic `azapi_resource` takes a `type`
(e.g. `Microsoft.AppConfiguration/configurationStores@2024-05-01`) and a JSON `body`, and azapi
PUT/PATCH/GETs it against ARM.

It does **not** depend on go-azure-sdk; it uses the generic Azure core client and Plugin Framework:

```
github.com/Azure/azure-sdk-for-go/sdk/azcore v1.21.1
github.com/Azure/azure-sdk-for-go/sdk/azidentity v1.13.1
github.com/hashicorp/terraform-plugin-framework v1.19.0
```
([`go.mod` @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/go.mod))

### The embedded type schema

To validate and shape arbitrary resource bodies without hand-coded schemas, azapi **embeds a complete
ARM type catalogue**. `internal/azure/loader.go` embeds the `generated/` directory and loads an index
at runtime:

```go
//go:embed generated
var StaticFiles embed.FS

func GetAzureSchema() *Schema {
    ...
    data, err := StaticFiles.ReadFile("generated/index.json")
    ...
}
```
([`internal/azure/loader.go` @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/internal/azure/loader.go))

The embedded tree is keyed by service / `microsoft.<service>` / API version, with one `types.json`
per version, e.g.
`internal/azure/generated/appconfiguration/microsoft.appconfiguration/2024-05-01/types.json`
([leaf dir @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/tree/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/internal/azure/generated/appconfiguration/microsoft.appconfiguration/2024-05-01)).
This is what lets azapi be **schema-driven**: given a `type@version`, it looks up the embedded type
definition and uses it for validation/plan-shaping rather than a hand-written Go schema.

### Where the types come from (and the shared source)

azapi does not generate these types itself — it **copies them from the Bicep types project**. The
update script removes the existing `internal/azure/generated` and copies a Bicep `generated/`
directory in:

```sh
# scripts/bicep-types-update.sh
rm -r "$ROOTDIR/internal/azure/generated"
cp -r "$bicep_dir/generated" "$ROOTDIR/internal/azure/generated"
find . -name "*.md" -type f -delete
```
([`scripts/bicep-types-update.sh` @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/scripts/bicep-types-update.sh))

The source is `Azure/bicep-types-az` (azapi historically tracked the `ms-henglu/bicep-types-az` fork
in its CHANGELOG, and references `Azure/bicep-types-az` issues directly in code, e.g.
`internal/azure/types/integer_type.go`'s note pointing at `Azure/bicep-types-az/issues/1404`)
([code search @ efeb3ca](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/internal/azure/types/integer_type.go)).

And `bicep-types-az` is **itself generated from `Azure/azure-rest-api-specs`** via an AutoRest
extension (`src/autorest.bicep`): its README documents generating types from a local checkout of the
azure-rest-api-specs repo
([Azure/bicep-types-az](https://github.com/Azure/bicep-types-az)). `?? verify` a commit-pinned
permalink to the bicep-types-az README/generator line and the exact azapi `types.json` schema fields
(noted in DO_NEXT) — these were read via WebFetch/dir-listing rather than a pinned raw file.

So azapi's "schema" is, transitively, the same Microsoft Swagger that powers azurerm-via-pandora and
the official SDKs — just routed through the Bicep type-loader format instead of pandora's data model.

## The contrast, in one table

| | azurerm | go-azure-sdk | azapi |
|---|---|---|---|
| Authoring model | **Hand-written** resources (schema + CRUD), per resource | **Generated** API client | **Generic / schema-driven** — one resource type, no per-resource code |
| Framework | Plugin SDK v2 + Plugin Framework (muxed), migrating to typed/Framework | n/a (library) | Plugin Framework only |
| API client | calls into **go-azure-sdk** | (is the client) | generic `azcore`/`azidentity` against ARM |
| Generator | partly scaffolded by pandora `generator-terraform`; mostly hand-finished | **pandora** (`importer-rest-api-specs` → data-api → `generator-go-sdk`) | none for the provider; **embeds** Bicep-generated `types.json` |
| Spec source | (via go-azure-sdk) | `Azure/azure-rest-api-specs` Swagger (submodule) | `azure-rest-api-specs` → `bicep-types-az` → embedded types |
| License | MPL-2.0 | MPL-2.0 (+ MIT Microsoft-derived code) | MPL-2.0 |

The recurring theme: **`Azure/azure-rest-api-specs` is the common upstream.** Microsoft's AutoRest/
TypeSpec generate `azure-sdk-for-*` from it; HashiCorp's pandora generates `go-azure-sdk` from it;
and the Bicep `autorest.bicep` extension generates the type catalogue that azapi embeds from it.
