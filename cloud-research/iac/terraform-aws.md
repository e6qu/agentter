# AWS Terraform Providers: Two Build Models, One Cloud

HashiCorp ships **two** Terraform providers for AWS, built on opposite
philosophies:

| | `terraform-provider-aws` | `terraform-provider-awscc` |
|---|---|---|
| Build model | **Mostly hand-written Go**, with a code-gen system for scaffolding | **Fully generated** from CloudFormation resource schemas |
| Runtime API | AWS SDK for Go v2 (per-service clients) | AWS **Cloud Control API** (one generic client) |
| TF plugin layer | Plugin SDK v2 (legacy) + Plugin Framework (migrating) | Plugin Framework only |
| Resource count | ~1,400 resources, each hand-authored | ~1,366 resources, each machine-emitted |
| Spec source | AWS SDK service models + a hand-maintained service registry | `aws-cloudformation-resource-schema` JSON (same source AWS CDK L1 uses) |
| License | MPL-2.0 | MPL-2.0 |

Pinned commits used throughout this note:

- `terraform-provider-aws` @ `e26a14e0d078679494673567ab79363a7c1a60a0`
- `terraform-provider-awscc` @ `e9fd27e90f70843339c5a6a4296e8e78282e69c6`

## 1. `terraform-provider-aws` — hand-written, with a code-gen scaffolding layer

The mainstream provider is **not** generated. Each resource's schema and its
Create/Read/Update/Delete logic are written by hand in Go on top of the AWS SDK
for Go v2. What *is* generated is the surrounding **scaffolding**: service
registration, tag helpers, paginators, endpoint resolvers, identity/tagging
tests, doc lists, etc. These generators live under
[`internal/generate/`](https://github.com/hashicorp/terraform-provider-aws/tree/e26a14e0d078679494673567ab79363a7c1a60a0/internal/generate)
and are driven by per-service `//go:generate` directives.

### 1.1 The `internal/generate/` toolbox

`internal/generate/` contains ~30 single-purpose Go generators. The
load-bearing ones:

- **`servicepackage/`** — emits each service's
  `service_package_gen.go` (the resource/data-source/ephemeral registration
  table) by scanning hand-written files for `// @FrameworkResource` /
  `// @SDKResource` annotations.
- **`tags/`** — emits per-service `tags_gen.go` (ListTags / UpdateTags /
  ServiceTagsMap, etc.), parameterized by flags that describe each service's
  tagging API shape.
- **`listpages/`** — emits `list_pages_gen.go` pagination wrappers for SDK
  operations that AWS didn't ship a paginator for.
- **`tagstests/`, `identitytests/`, `serviceendpointtests/`,
  `sweeperregistration/`** — emit test and sweeper boilerplate.
- **`namesconsts/`, `namescapslist/`, `servicepackages/`** — emit
  cross-service constant lists and the master registration list.

Everything is anchored by a single hand-maintained registry,
[`names/data/names_data.hcl`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/names/data/names_data.hcl),
which maps each service to its AWS SDK ID, ARN namespace, provider package
name, and resource prefix:

```hcl
service "accessanalyzer" {
  sdk {
    id            = "AccessAnalyzer"
    arn_namespace = "access-analyzer"
  }
  names {
    provider_name_upper = "AccessAnalyzer"
    human_friendly      = "IAM Access Analyzer"
  }
  ...
}
```

### 1.2 How generation is wired: per-service `generate.go`

Each service package has a tiny `generate.go` whose only job is to hold
`//go:generate` lines. For CloudWatch Logs,
[`internal/service/logs/generate.go`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/internal/service/logs/generate.go):

```go
//go:generate go run ../../generate/listpages/main.go -ListOps=DescribeAccountPolicies,DescribeIndexPolicies,DescribeQueryDefinitions,DescribeResourcePolicies
//go:generate go run ../../generate/tags/main.go -ListTags -ServiceTagsMap -UpdateTags -CreateTags -KVTValues
//go:generate go run ../../generate/servicepackage/main.go
//go:generate go run ../../generate/tagstests/main.go
//go:generate go run ../../generate/identitytests/main.go
// ONLY generate directives and package declaration! Do not add anything else to this file.
```

Running `go generate` over the tree re-emits all the `*_gen.go` files.

### 1.4 Hand-written vs generated: the line

The **resource itself is hand-written**; the generator only reads its
annotations and emits the registration row. In
[`internal/service/logs/anomaly_detector.go`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/internal/service/logs/anomaly_detector.go):

```go
// @FrameworkResource("aws_cloudwatch_log_anomaly_detector", name="Anomaly Detector")
// @Tags(identifierAttribute="arn")
// @Testing(importStateIdFunc="testAccAnomalyDetectorImportStateIDFunc")
func newAnomalyDetectorResource(context.Context) (resource.ResourceWithConfigure, error) {
	r := &anomalyDetectorResource{}
	return r, nil
}
```

The schema, the `Create`/`Read`/`Update`/`Delete` methods, the flatten/expand
logic — all hand-authored below this. The generator never writes them; it only
consumes the `@`-comments to wire the resource into the provider and to
generate matching tests.

### 1.5 Plugin SDK v2 → Plugin Framework migration

The provider is mid-migration from the legacy **Terraform Plugin SDK v2** to
the **Terraform Plugin Framework**, and both styles coexist *within the same
service package*. CloudWatch Logs shows both:

- **Plugin SDK v2 (legacy):**
  [`internal/service/logs/group.go`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/internal/service/logs/group.go)
  uses `// @SDKResource("aws_cloudwatch_log_group", ...)`, returns a
  `*schema.Resource`, and implements `resourceGroupCreate/Read/Update/Delete`
  with `*schema.ResourceData`.
- **Plugin Framework (new):** `anomaly_detector.go` above uses
  `// @FrameworkResource(...)` and the framework `resource.Resource`
  interface.

The annotation type (`@SDKResource` vs `@FrameworkResource`) is what tells the
`servicepackage` generator which registration list to add the resource to
(`SDKResources(...)` vs `FrameworkResources(...)`). New resources are expected
to use the Framework; the legacy hand-written `group.go` even carries a
`// DONOTCOPY` banner pointing authors at the `skaff` scaffolding tool
([`skaff/`](https://github.com/hashicorp/terraform-provider-aws/tree/e26a14e0d078679494673567ab79363a7c1a60a0/skaff)).

## 2. `terraform-provider-awscc` — fully generated from CloudFormation schemas

The AWS Cloud Control provider is the **pure-codegen** AWS provider. There is
no hand-written per-resource logic at all: a generator reads CloudFormation
resource type schemas and emits Terraform **Plugin Framework** resources for
all ~1,366 supported types. At the pinned commit there are **1366**
`*_resource_gen.go` files under
[`internal/aws/`](https://github.com/hashicorp/terraform-provider-awscc/tree/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/aws).
The provider's own README states it plainly:

> This provider is generated from the latest CloudFormation schemas, and will
> release weekly containing all new services and enhancements added to Cloud
> Control. ([README.md](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/README.md))

### 2.1 The source schemas (== AWS CDK L1's source)

The generator's input is the **CloudFormation resource type schema** JSON — the
same `aws-cloudformation-resource-schema` documents that AWS CDK's L1 (`Cfn*`)
constructs are generated from. The provider parses them with HashiCorp's
[`github.com/hashicorp/aws-cloudformation-resource-schema-sdk-go`](https://github.com/hashicorp/aws-cloudformation-resource-schema-sdk-go)
(`v0.23.0`, per
[`go.mod`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/go.mod)),
imported as `cfschema` in the emitter.

Schemas are downloaded from the CloudFormation registry and cached in-repo
under
[`internal/service/cloudformation/schemas/`](https://github.com/hashicorp/terraform-provider-awscc/tree/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/service/cloudformation/schemas)
(e.g.
[`AWS_Logs_AccountPolicy.json`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/service/cloudformation/schemas/AWS_Logs_AccountPolicy.json)):

```json
{
  "typeName": "AWS::Logs::AccountPolicy",
  "description": "The AWS::Logs::AccountPolicy resource specifies a CloudWatch Logs AccountPolicy.",
  "sourceUrl": "https://github.com/aws-cloudformation/aws-cloudformation-resource-providers-logs.git",
  "properties": {
    "PolicyName": { "type": "string", "minLength": 1, "maxLength": 256, "pattern": "^[^:*]{1,256}$" },
    "PolicyDocument": { "type": "string", "minLength": 1, "maxLength": 30720 },
    ...
  }
}
```

> **Relationship to AWS CDK:** awscc and AWS CDK consume the *same upstream
> artifact*. CDK uses these CloudFormation resource type schemas to generate
> its L1 `Cfn*` constructs; awscc uses them to generate Terraform Plugin
> Framework resources. Same input spec, two different SDK/IaC outputs.

### 2.2 The generation pipeline

Three generators chain together, all under
[`internal/provider/generators/`](https://github.com/hashicorp/terraform-provider-awscc/tree/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators):

1. **`allschemas/`** — calls the Cloud Control API to enumerate all
   CloudFormation types available, emitting a dated HCL manifest
   (e.g.
   [`available_schemas.2026-05-27.hcl`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators/allschemas/available_schemas.2026-05-27.hcl),
   whose header reads `# 1390 CloudFormation resource types schemas are available`).
   Each entry maps a Terraform type to a CloudFormation type:
   ```hcl
   resource_schema "aws_logs_account_policy" {
     cloudformation_type_name = "AWS::Logs::AccountPolicy"
   }
   ```

2. **`schema/`** — the "schema downloader". Reads the curated
   `all_schemas.hcl` config, downloads/caches each CloudFormation schema,
   verifies it against the CloudFormation meta-schema, and **emits a
   `//go:generate` driver file** that invokes the resource generator once per
   resource. That driver is
   [`internal/provider/resources.go`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/resources.go):
   ```go
   // Code generated by generators/schema/main.go; DO NOT EDIT.
   //go:generate go run generators/resource/main.go -resource awscc_logs_account_policy -cfschema ../service/cloudformation/schemas/AWS_Logs_AccountPolicy.json -package logs -- ../aws/logs/account_policy_resource_gen.go ../aws/logs/account_policy_resource_gen_test.go
   ```
   Triggered from
   [`internal/provider/schemas.go`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/schemas.go):
   ```go
   //go:generate go run generators/schema/main.go -config all_schemas.hcl -generated-code-root .. ... -- resources.go singular_data_sources.go plural_data_sources.go import_examples_gen.json
   ```

3. **`resource/`** (+ `singular-data-source/`, `plural-data-source/`) — the
   per-resource emitter. For each `-cfschema <file>` it walks the schema
   property tree via the shared
   [`codegen.Emitter`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators/shared/codegen/emitter.go)
   — switching on `cfschema.PropertyTypeString/Integer/Array/Object/...` — and
   renders a Plugin Framework schema through
   [`resource/schema.tmpl`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators/resource/schema.tmpl).

### 2.3 What a generated resource looks like (and its header)

Header + body of
[`internal/aws/logs/account_policy_resource_gen.go`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/aws/logs/account_policy_resource_gen.go):

```go
// Code generated by generators/resource/main.go; DO NOT EDIT.

package logs

func init() {
	registry.AddResourceFactory("awscc_logs_account_policy", accountPolicyResource)
}

// accountPolicyResource returns the Terraform awscc_logs_account_policy resource.
// This Terraform resource corresponds to the CloudFormation AWS::Logs::AccountPolicy resource.
func accountPolicyResource(ctx context.Context) (resource.Resource, error) {
	attributes := map[string]schema.Attribute{ /*START SCHEMA*/
		// Property: PolicyDocument
		// CloudFormation resource type schema:
		//	{
		//	  "maxLength": 30720,
		//	  "minLength": 1,
		//	  "type": "string"
		//	}
		"policy_document": schema.StringAttribute{
			Required:   true,
			Validators: []validator.String{stringvalidator.LengthBetween(1, 30720)},
		},
		...
	}
	...
	opts = opts.WithCloudFormationTypeName("AWS::Logs::AccountPolicy").WithTerraformTypeName("awscc_logs_account_policy")
	opts = opts.WithTerraformSchema(schema)
	opts = opts.WithPrimaryIdentifier(
		identity.Identifier{Name: "account_id", RequiredForImport: true},
		identity.Identifier{Name: "policy_type", RequiredForImport: true},
		identity.Identifier{Name: "policy_name", RequiredForImport: true})
	opts = opts.WithAttributeNameMap(map[string]string{
		"account_id": "AccountId", "policy_document": "PolicyDocument", ...
	})
	v, err := generic.NewResource(ctx, opts...)
	...
}
```

Two notable design choices visible here:

- The **original CloudFormation schema fragment is embedded verbatim as a
  comment** above each attribute — the generated file is self-documenting and
  trivially diffable against the upstream schema.
- The generated code contains **no CRUD logic**. It only declares the schema,
  the CloudFormation↔Terraform name map, and the primary identifier, then hands
  off to `generic.NewResource`.

### 2.4 The generic Cloud Control runtime

All ~1,366 generated resources delegate to one runtime,
[`internal/generic/resource.go`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/generic/resource.go),
which translates Terraform state to/from JSON and calls the AWS **Cloud Control
API** (`github.com/aws/aws-sdk-go-v2/service/cloudcontrol`) generically:

```go
input := &cloudcontrol.CreateResourceInput{ ... }
output, err := conn.CreateResource(ctx, input)
```

Because Cloud Control offers uniform CRUDL across every CloudFormation-registered
type, a single generic resource implementation plus a per-type generated schema
is sufficient — no service-specific Go is needed.

## 3. Same cloud, opposite philosophies

- **`terraform-provider-aws`**: per-service AWS SDK v2 clients; resources and
  their CRUD are hand-written for fidelity and ergonomics; code-gen only
  produces *scaffolding* (registration, tags, paginators, tests) from
  annotations and a hand-maintained service registry. Mid-migration Plugin SDK
  v2 → Plugin Framework.
- **`terraform-provider-awscc`**: one generic Cloud Control client; *every*
  resource is machine-emitted from CloudFormation resource type schemas; zero
  hand-written per-resource logic; Plugin Framework only; weekly releases that
  simply re-run the generator over the latest schemas.

The shared insight: awscc's input — the CloudFormation resource type schemas —
is the **same artifact AWS CDK's L1 constructs are generated from**. awscc is,
in effect, "CDK L1 for Terraform," whereas terraform-provider-aws is the
artisanal, hand-tuned provider that trades generation breadth for per-resource
quality.
