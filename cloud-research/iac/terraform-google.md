# Google Terraform Provider Generation via Magic Modules

How `GoogleCloudPlatform/magic-modules` generates the Google Cloud Terraform
providers — `hashicorp/terraform-provider-google` (GA) and
`hashicorp/terraform-provider-google-beta` — from a single set of resource
definitions.

All permalinks are pinned to the following commits (HEAD as of 2026-05-30):

- magic-modules: `8cc65df4a76512da5e02abe7fcfd39657374b906`
- terraform-provider-google: `e57828c42bad77882b4cadd55687b9c9fb6db220`
- terraform-provider-google-beta: `60823385e868cafc478d1e25f3c3c36274ff7ee0`

---

## 1. The big picture

Magic Modules ("MM") is a code generator that maps a Google Cloud REST API
resource to a Terraform resource/datasource. The same source of truth feeds
**multiple downstream providers** — primarily `terraform-provider-google` (GA)
and `terraform-provider-google-beta` — and historically other downstreams
(Ansible, InSpec, the Terraform Validator / `terraform-google-conversion`).
The Google Terraform provider is one of the most thoroughly code-generated
cloud-tooling artifacts in existence: the overwhelming majority of resources
in both providers are emitted from MM, and the generated files even carry a
machine-readable header pointing back to the exact YAML config and template
that produced them (see §2).

There are (now) two layers of resources:

1. **MMv1** — declarative YAML resource definitions under
   `mmv1/products/<product>/<Resource>.yaml`, rendered through Go
   `text/template` templates in `mmv1/templates/terraform/`. This is the
   primary, actively-grown path.
2. **Handwritten** — Go resources/services that MM copies verbatim (with
   light templating) from `mmv1/third_party/terraform/` into the output
   provider. Used where the API surface is too irregular for MMv1 (e.g.
   `google_storage_bucket`, `google_compute_instance`).

A third historical path — **DCL-based** resources generated by `tpgtools`
on top of Google's Declarative Client Library — **was removed from
magic-modules' `main` on 2026-05-18** (see §4). It is no longer part of the
pipeline; remaining DCL resources are being migrated back to MMv1.

## 2. MMv1: YAML resource definitions → generated Go

### A product YAML

`mmv1/products/pubsub/Topic.yaml` describes the entire API surface of
`google_pubsub_topic` declaratively — CRUD verbs/URLs, async polling, IAM,
custom code hooks, and field schema:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/pubsub/Topic.yaml#L14-L56>

```yaml
name: 'Topic'
description: |
  A named resource to which messages are sent by publishers.
base_url: 'projects/{{project}}/topics'
create_verb: 'PUT'
update_url: 'projects/{{project}}/topics/{{name}}'
update_verb: 'PATCH'
update_mask: true
async:
  type: 'PollAsync'
  check_response_func_existence: 'transport_tpg.PollCheckForExistence'
  check_response_func_absence: 'transport_tpg.PollCheckForAbsence'
iam_policy:
  method_name_separator: ':'
  parent_resource_attribute: 'topic'
custom_code:
  encoder: 'templates/terraform/encoders/no_send_name.go.tmpl'
  update_encoder: 'templates/terraform/update_encoder/pubsub_topic.tmpl'
```

Fields are declared under `properties:`
(<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/pubsub/Topic.yaml#L128-L153>):

```yaml
properties:
  - name: 'name'
    type: String
    description: 'Name of the topic.'
    required: true
    immutable: true
    diff_suppress_func: 'tpgresource.CompareSelfLinkOrResourceName'
    custom_flatten: 'templates/terraform/custom_flatten/name_from_self_link.tmpl'
    custom_expand: 'templates/terraform/custom_expand/resource_from_self_link.go.tmpl'
  - name: 'kmsKeyName'
    type: String
    description: |
      The resource name of the Cloud KMS CryptoKey ...
  - name: 'labels'
    type: KeyValueLabels
    description: |
      A set of key/value label pairs to assign to this Topic.
```

Each property carries the Terraform schema metadata (`required`, `immutable`,
`diff_suppress_func`) and optional escape hatches into hand-written Go snippets
(`custom_flatten`, `custom_expand`) which themselves are template files in
`mmv1/templates/terraform/custom_*`.

### The template

The principal resource template is **`mmv1/templates/terraform/resource.go.tmpl`**
— a Go `text/template` (not ERB). Its top emits the generated-file header via
the `$.CodeHeader` method and then the Go package/imports:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/templates/terraform/resource.go.tmpl#L1-L21>

```gotemplate
{{$.CodeHeader TemplatePath}}

package {{ lower $.ProductMetadata.Name }}

import (
    "bytes"
    "context"
    ...
```

`CodeHeader` is a Go method on the `Resource` type that builds the
"AUTO GENERATED CODE … Magic Modules" banner, embedding the URL of the source
YAML (`r.GithubURL()`) and the template path:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/api/resource.go#L2602-L2619>

```go
func (r Resource) CodeHeader(templatePath string) string {
	templateUrl := GITHUB_BASE_URL + templatePath
	return fmt.Sprintf(`// ...
//     ***     AUTO GENERATED CODE    ***    Type: MMv1     ***
// ...
//     Configuration: %s
//     Template:      %s
//
//     DO NOT EDIT this file directly. ...`, r.GithubURL(), templateUrl)
}
```

`GITHUB_BASE_URL` is hard-coded to the magic-modules tree:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/api/resource.go#L2602> (the constant is
`"https://github.com/GoogleCloudPlatform/magic-modules/tree/main/" + RELATIVE_MAGICIAN_LOCATION`).

### The generated output (header naming magic-modules)

The `Topic.yaml` above produces
`google/services/pubsub/resource_pubsub_topic.go` in the **GA** provider, whose
header points straight back to the YAML and template that generated it:
<https://github.com/hashicorp/terraform-provider-google/blob/e57828c42bad77882b4cadd55687b9c9fb6db220/google/services/pubsub/resource_pubsub_topic.go#L1-L20>

```go
// ----------------------------------------------------------------------------
//
//     ***     AUTO GENERATED CODE    ***    Type: MMv1     ***
//
// ----------------------------------------------------------------------------
//
//     This code is generated by Magic Modules using the following:
//
//     Configuration: https://github.com/GoogleCloudPlatform/magic-modules/tree/main/mmv1/products/pubsub/Topic.yaml
//     Template:      https://github.com/GoogleCloudPlatform/magic-modules/tree/main/mmv1/templates/terraform/resource.go.tmpl
//
//     DO NOT EDIT this file directly. Any changes made to this file will be
//     overwritten during the next generation cycle.
//
// ----------------------------------------------------------------------------
```

The **beta** provider's
`google-beta/services/pubsub/resource_pubsub_topic.go` carries the *identical*
header pointing to the *same* `Topic.yaml` — concretely demonstrating one source
of truth feeding two downstreams:
<https://github.com/hashicorp/terraform-provider-google-beta/blob/60823385e868cafc478d1e25f3c3c36274ff7ee0/google-beta/services/pubsub/resource_pubsub_topic.go#L4-L18>

### Engine state: Ruby → Go migration `?? verify` (resolved)

Historically MM was a Ruby program (`mmv1/compiler.rb`,
`mmv1/provider/terraform.rb`, ERB `.erb` templates under `mmv1/compile/`). It
has since been rewritten in Go:

- `mmv1/` is now a Go module with `go.mod`, `main.go`, and `cmd/`:
  <https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1>
  The Go replacements for `compiler.rb` / `provider/terraform.rb` are
  `mmv1/main.go`
  (<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/main.go#L1-L19>)
  and `mmv1/provider/terraform.go`
  (<https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/provider>).
- Templates are now Go `text/template` `.tmpl` files; there are **no `.erb`
  files** in the tree and **no `mmv1/compile/` directory** (both return 404 at
  the pinned commit).
- The very last Ruby remnants were deleted on **2026-05-18** in PR #17608
  "Removed trailing ruby-related code and tpgtools" (15,415 deletions):
  <https://github.com/GoogleCloudPlatform/magic-modules/pull/17608>

So the engine is now **entirely Go**; the Ruby→Go migration is complete as of
mid-May 2026. (Background on the migration:
<https://googlecloudplatform.github.io/magic-modules/reference/ruby-go-changes/>.)

## 3. Handwritten overrides and the generated-vs-handwritten split

Not every resource is MMv1-generated. Two override mechanisms exist:

### `mmv1/third_party/terraform/` (whole hand-written resources/services)

This directory holds hand-authored Go that MM copies into the output provider
(with light Go-template substitution for GA/beta differences — note the
`.go.tmpl` suffix):
<https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/third_party/terraform>

The canonical example is **`google_storage_bucket`**. Its MMv1 YAML
(`storage/Bucket.yaml`) sets `exclude_resource: true`, telling MM *not* to
generate the resource (only docs/datasource scaffolding):
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/storage/Bucket.yaml#L14-L27>

```yaml
name: 'Bucket'
kind: 'storage#bucket'
description: |
  The Buckets resource represents a bucket in Google Cloud Storage. ...
exclude_resource: true
```

The actual resource is hand-written at
`mmv1/third_party/terraform/services/storage/resource_storage_bucket.go.tmpl`:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/third_party/terraform/services/storage/resource_storage_bucket.go.tmpl#L1-L8>

In the output provider it lands as
`google/services/storage/resource_storage_bucket.go`, whose header marks it
`Type: Handwritten` and cites the *source `.tmpl`* (not a YAML config):
<https://github.com/hashicorp/terraform-provider-google/blob/e57828c42bad77882b4cadd55687b9c9fb6db220/google/services/storage/resource_storage_bucket.go#L1-L16>

```go
//     ***     AUTO GENERATED CODE    ***    Type: Handwritten     ***
// ...
//     Source file: https://github.com/GoogleCloudPlatform/magic-modules/tree/main/mmv1/third_party/terraform/services/storage/resource_storage_bucket.go.tmpl
//
//     DO NOT EDIT this file directly. ...
```

So the **header `Type:` field distinguishes the split** at a glance:
`Type: MMv1` (generated from YAML) vs `Type: Handwritten` (copied from
`third_party/`). `third_party/terraform/` also supplies the non-resource
plumbing every provider needs: `provider/`, `transport/`, `tpgresource/`,
`fwprovider/` (plugin-framework), `acctest/`, `verify/`, `version/`, the
provider `main.go.tmpl`, etc.
<https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/third_party/terraform>

### Per-field custom code hooks

Even within an MMv1 resource, individual behaviors can drop into hand-written Go
via `custom_code:` (resource-level: `encoder`, `update_encoder`, `decoder`,
`pre_create`, `post_read`, …) or per-property `custom_expand` / `custom_flatten`
/ `diff_suppress_func` — visible in the `Topic.yaml` excerpts in §2. The
referenced snippets live under `mmv1/templates/terraform/{encoders,custom_expand,
custom_flatten,...}/`.

## 4. The (removed) DCL / `tpgtools` path

A second generator, **`tpgtools`**, formerly lived at the repo root
(`magic-modules/tpgtools`). It generated "DCL-based" resources from Google's
**Declarative Client Library (DCL)**, which exposes OpenAPI schema objects per
resource; `tpgtools` turned those into Terraform resources. It was conceived as
a lighter-weight "MMv2" running side-by-side with MMv1, but the provider team
ultimately decided to migrate DCL resources back to MMv1.

At the pinned commit `tpgtools` **no longer exists** (the path 404s), having
been deleted together with the trailing Ruby code in PR #17608
"Removed trailing ruby-related code and tpgtools" (merged 2026-05-18):
<https://github.com/GoogleCloudPlatform/magic-modules/pull/17608>
(commit `9ab8dabbe7…`). So as of HEAD, MM has a **single active engine (MMv1,
in Go) plus hand-written `third_party/` code**; the DCL/tpgtools path is
historical. `?? verify` — the exact set of resources still mid-migration from
DCL to MMv1 was not enumerated here.

## 5. Output repos

- **`hashicorp/terraform-provider-google`** (GA) — generated files carry the
  `Type: MMv1` / `Type: Handwritten` "Magic Modules" header naming the upstream
  YAML/template (§2, §3). Service code lives under `google/services/<product>/`.
  License: **Mozilla Public License 2.0**
  (<https://github.com/hashicorp/terraform-provider-google/blob/e57828c42bad77882b4cadd55687b9c9fb6db220/LICENSE#L1-L3>;
  files also carry `// SPDX-License-Identifier: MPL-2.0`).
- **`hashicorp/terraform-provider-google-beta`** — same generated headers,
  same upstream YAML, code under `google-beta/services/<product>/` (§2).

Both are **downstream mirrors**: humans do not hand-edit generated files in
them (the header literally says "DO NOT EDIT this file directly"). MM opens
synchronized PRs to both.

### Licensing nuance (generator vs copied code)

magic-modules itself is **dual-licensed**: its `LICENSE` splits the tree —
`mmv1/third_party/terraform/*` and `tools/go-changelog/*` are **MPL-2.0**
(so the hand-written code copied into the providers matches the providers'
MPL-2.0 license), while **everything else (the generator) is Apache-2.0**:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/LICENSE#L1-L2>
(MPL section) and
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/LICENSE#L378-L381>
(Apache section, `Files: *, excluding tools/go-changelog/* and mmv1/third_party/terraform/*`).
Product YAMLs themselves carry an **Apache-2.0** header
(<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/pubsub/Topic.yaml#L1-L12>),
the resource template a Google **Apache-2.0** header
(<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/templates/terraform/resource.go.tmpl#L1-L14>).

## 6. How a contributor changes a YAML def → regenerates both providers

The generation is driven by the repo's `GNUmakefile`, which builds the Go
`mmv1` binary and runs it once per version against the *same* `products/` YAML:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/GNUmakefile#L43-L60>

```make
mm_binary:
	cd mmv1 && go build -o $(MM_BINARY) .

mmv1: mm_binary
	cd mmv1; \
	if [ "$(VERSION)" = "ga" ]; then \
		$(MM_BINARY) --output $(OUTPUT_PATH) --version ga --no-docs $(mmv1_args) \
		&& $(MM_BINARY) --output $(OUTPUT_PATH) --version beta --no-code $(mmv1_args); \
	else \
		$(MM_BINARY) --output $(OUTPUT_PATH) --version $(VERSION) $(mmv1_args); \
	fi
```

The `--version` flag on `mmv1/main.go` selects the channel; resources/fields
gated with `min_version: 'beta'` in YAML are emitted only into the beta
provider:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/main.go#L27-L29>
(the `--version` flag) and an example field-level gate in
`compute/Subnetwork.yaml`:
<https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/compute/Subnetwork.yaml#L188>.

The contributor loop (per the docs
<https://googlecloudplatform.github.io/magic-modules/develop/generate-providers/>):

1. Edit `mmv1/products/<product>/<Resource>.yaml` (add a field, change a CRUD
   URL, add an example) — or, for handwritten resources, edit the file under
   `mmv1/third_party/terraform/...`.
2. Regenerate the GA provider:
   ```
   make provider VERSION=ga   OUTPUT_PATH="$GOPATH/src/github.com/hashicorp/terraform-provider-google"
   ```
3. Regenerate the beta provider:
   ```
   make provider VERSION=beta OUTPUT_PATH="$GOPATH/src/github.com/hashicorp/terraform-provider-google-beta"
   ```
   (`make provider` cleans the output dir, runs the `mmv1` binary, and the
   provider code is rewritten in place.)
4. Open a single PR against magic-modules. CI generates "downstream" diffs for
   both providers (and TGC), runs tests, and on merge the bot pushes the
   generated changes to the two downstream repos.

A single YAML edit therefore lands, consistently, in *both* `google` and
`google-beta` — the defining property of the Magic Modules pipeline.

## 7. Summary

- magic-modules is a **Go** generator (`mmv1/`, Go `text/template` `.tmpl`
  templates) that emits the GA and beta Google Terraform providers from one set
  of `mmv1/products/<product>/<Resource>.yaml` definitions. The Ruby/ERB engine
  and the DCL/`tpgtools` generator were **both removed on 2026-05-18**
  (PR #17608); the pipeline is now single-engine MMv1 + hand-written
  `third_party/` code.
- Generated files in both providers carry an `AUTO GENERATED CODE … Magic
  Modules` header naming the exact upstream YAML config and template
  (`Type: MMv1`) or the upstream `.go.tmpl` (`Type: Handwritten`) — making the
  generated-vs-handwritten split self-documenting.
- The split: most resources are MMv1-generated; irregular ones (e.g.
  `google_storage_bucket`) set `exclude_resource: true` in YAML and are
  hand-written under `mmv1/third_party/terraform/services/<product>/`, plus all
  provider plumbing (transport, provider entrypoint, plugin-framework) lives in
  `third_party/`.
- Licensing: generator/configs are **Apache-2.0**; the hand-written
  `third_party/terraform/*` copied into the providers is **MPL-2.0**, matching
  the providers' own MPL-2.0 license.

### Gaps / `?? verify`

- The precise list of resources still mid-migration from DCL→MMv1 after
  tpgtools' removal was not enumerated.
- The CI machinery that turns one magic-modules PR into synchronized downstream

- Field-level vs example-level `min_version: 'beta'` semantics: cited an
