# Oracle Cloud Infrastructure (OCI) — SDK Code Generation Research Notes

> Evidence basis: OCI's code generator is **not open source**. Almost all evidence
> here is *indirect* — derived from autogeneration headers, version stamps, and
> repo structure in the **published** SDK repos. Each claim below is tagged
> **[EVIDENCE]** (directly verified from a file/commit) or **[INFERENCE]**
> (reasoned from structure). `?? verify:` marks open gaps.

## 1. Service description / IDL

**Finding: OCI describes each service with a date-versioned OpenAPI/Swagger 2.0
spec. The spec is the codegen input. Specs are NOT published as raw files in the
public SDK repos; the public-facing form is the rendered API reference on
docs.oracle.com.**

- **[EVIDENCE]** The TypeScript generated client header literally states the IDL
  format and version: `* OpenAPI spec version: 20160918` for Object Storage.
  See `lib/objectstorage/lib/client.ts` header:
  https://github.com/oracle/oci-typescript-sdk/blob/8c11d0c62800f284cc12d59cb9052518d1f9bea1/lib/objectstorage/lib/client.ts#L1-L14
  (header quotes: "Object Storage Service API ... OpenAPI spec version: 20160918
  ... NOTE: This class is auto generated by OracleSDKGenerator.")

- **[EVIDENCE]** The spec version is **date-based** and **per-service**, embedded
  in every generated client across languages:
  - Object Storage → `20160918`
    (Python: https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/object_storage/object_storage_client.py#L5 ;
    Go header at https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/objectstorage/objectstorage_client.go#L1-L3 ;
    Java `@jakarta.annotation.Generated(... comments = "API Version: 20160918")`
    at https://github.com/oracle/oci-java-sdk/blob/013bb11dca68f84ec10a8e48e99855dfbb3a8f13/bmc-objectstorage/bmc-objectstorage-generated/src/main/java/com/oracle/bmc/objectstorage/ObjectStorageClient.java#L15 )
  - Functions / ONS → `20181201`
    (Python functions invoke: https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/functions/functions_invoke_client.py#L5 ;
    TS functions: https://github.com/oracle/oci-typescript-sdk/blob/8c11d0c62800f284cc12d59cb9052518d1f9bea1/lib/functions/lib/client.ts )
  - Queue → `20210201`
    (Python queue admin: https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/queue/queue_admin_client.py#L5 )

  These dates are the OCI "API version" date convention (the date the service's
  API contract was first cut). The *same date* appears identically in every
  language's generated client — **[INFERENCE]** strongly implies one shared
  per-service spec drives all language generators.

- **[EVIDENCE]** No raw `swagger.json` / `openapi.json` spec digests are checked
  into the public SDK repos. GitHub code search for `swagger extension:json` in
  `oci-go-sdk`, `oci-python-sdk`, `oci-typescript-sdk` returns nothing. The
  embedded "spec" is only the *version string* in headers, not the document.

- **[EVIDENCE]** OCI does render the API contract publicly as an interactive API
  reference (date-versioned, e.g. `objectstorage/20160918`) at
  https://docs.oracle.com/en-us/iaas/api/ . OCI release notes confirm OCI works
  in terms of "OpenAPI (previously Swagger)" descriptions:
  https://docs.oracle.com/en-us/iaas/releasenotes/changes/87fee407-f103-4246-af19-dcd7b0a852c4/

- **?? verify:** The docs.oracle.com `/iaas/api/` portal is a JS-rendered SPA
  (WebFetch cannot read it). Whether a downloadable raw spec file is served from
  a backing endpoint (e.g. a condensed `.spec`/`.json`) is not directly confirmed
  here. The **source-of-truth spec lives internally** at Oracle and is not in a
  public `oracle/oci-api-spec` repo (no such repo exists — 404).

- **[INFERENCE]** Contact email in the TS/Java spec metadata
  (`opc_casper_users_us_grp@oracle.com`, visible in the objectstorage client
  header) reflects internal Oracle team ownership of the spec ("casper" is the
  internal codename for Object Storage). This is consistent with an internal
  spec registry.

## 2. Generator tool(s)

**Finding: One internal, closed-source generator named `OracleSDKGenerator`
produces all language SDKs (and the CLI). It is never published.**

- **[EVIDENCE]** The tool names itself in headers across every language:
  - Python: `# NOTE: This class is auto generated by OracleSDKGenerator. DO NOT
    EDIT. API Version: 20160918`
    https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/object_storage/object_storage_client.py#L5
  - TypeScript: `* NOTE: This class is auto generated by OracleSDKGenerator.`
    https://github.com/oracle/oci-typescript-sdk/blob/8c11d0c62800f284cc12d59cb9052518d1f9bea1/lib/objectstorage/lib/client.ts#L9-L11
  - Java: `@jakarta.annotation.Generated(value = "OracleSDKGenerator", comments
    = "API Version: 20160918")`
    https://github.com/oracle/oci-java-sdk/blob/013bb11dca68f84ec10a8e48e99855dfbb3a8f13/bmc-objectstorage/bmc-objectstorage-generated/src/main/java/com/oracle/bmc/objectstorage/ObjectStorageClient.java#L15
  - Go: uses a terser `// Code generated. DO NOT EDIT.` marker (no tool name on
    the service clients), e.g.
    https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/objectstorage/objectstorage_client.go#L1-L3

- **[EVIDENCE]** `OracleSDKGenerator` is the common tool across the whole SDK
  family. GitHub org-wide code search `OracleSDKGenerator org:oracle` returns
  ~1946 hits spanning `oci-typescript-sdk`, `oci-ruby-sdk`, `oci-dotnet-sdk`,
  `oci-powershell-modules`, and `oci-cli`. (Go/Java use different header text but
  the Java `@Generated(value="OracleSDKGenerator")` annotation proves the same
  tool underlies Java too.)

- **[EVIDENCE]** **No public artifact of the generator exists.** There is no
  `oracle/oci-sdk-codegen` / `oci-sdk-generator` repo; the headers are the only
  public trace. The generator binary/templates are internal.

- **[EVIDENCE — CLI]** `oci-cli` is **also generated by `OracleSDKGenerator`**,
  not merely transpiled from the Python SDK. The generated CLI command surface
  lives under per-service `.../generated/<service>_cli.py` with the same header:
  `# NOTE: This class is auto generated by OracleSDKGenerator. DO NOT EDIT. API
  Version: 20160918`
  https://github.com/oracle/oci-cli/blob/385f594d3bab0e5a17d9f97f6980b1772af6022e/services/object_storage/src/oci_cli_object_storage/generated/objectstorage_cli.py#L1-L4
  - **[INFERENCE]** The CLI is generated from the *same spec* (note identical
    `20160918`) but **imports the Python SDK `oci` as its runtime** (`import oci`
    in the same file). So: spec → generator → CLI command tree (click), with the
    Python SDK as the execution layer. The CLI is NOT a hand-coded wrapper.

- **[INFERENCE]** Go's `common/version.go` carries a *different* marker
  `// Code generated by go generate; DO NOT EDIT` — this is a local Go-toolchain
  generated version stamp, distinct from the OracleSDKGenerator service-client
  generation. https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/common/version.go#L1-L18

## 3. Pipeline / orchestration

**Finding: One monorepo per language; ~weekly synchronized releases with
identical feature sets across all languages — indicating one centralized,
batch-driven generation pipeline.**

- **[EVIDENCE — monorepo per language]**
  - Go: flat monorepo, one top-level dir per service
    (`objectstorage/`, `functions/`, `ons/`, `queue/`, ...):
    https://github.com/oracle/oci-go-sdk/tree/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368
  - Python: single package under `src/oci/<service>/`:
    https://github.com/oracle/oci-python-sdk/tree/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci
  - Java: multi-module Maven monorepo, one module per service with the legacy
    `bmc-` prefix (`bmc-objectstorage`, `bmc-functions`, `bmc-ons`, `bmc-queue`):
    https://github.com/oracle/oci-java-sdk/tree/013bb11dca68f84ec10a8e48e99855dfbb3a8f13
    ("bmc" = Bare Metal Cloud, OCI's original product name — a strong historical
    tell.)
  - TypeScript: monorepo with one package per service under `lib/<service>/`:
    https://github.com/oracle/oci-typescript-sdk/tree/8c11d0c62800f284cc12d59cb9052518d1f9bea1/lib

- **[EVIDENCE — Go Makefile reveals the generation contract]**
  `oci-go-sdk/Makefile` separates generated from hand-written targets and even
  contains the generator's template placeholder `##SPECNAME##`:
  https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/Makefile#L3-L4
  - Line 3: `GEN_TARGETS = identity core objectstorage ... functions ... ons ...
    queue ... ##SPECNAME##` — every service is a generated target; the trailing
    `##SPECNAME##` token is a literal codegen template marker that the internal
    pipeline appends new services into.
  - Line 4: `NON_GEN_TARGETS = common common/auth objectstorage/transfer example`
    — the hand-written surfaces are explicitly enumerated.
  - Lines 23-24: `# directories under gen targets which contains hand writen code`
    / `EXCLUDED_CLEAN_DIRECTORIES = objectstorage/transfer*` — the Object Storage
    transfer manager is hand-written code that the "clean generated code" step
    must skip.

- **[EVIDENCE — synchronized weekly cadence, identical content]** All four SDKs
  release on the **same dates** with **identical service feature lists**:
  - Go `65.116.0 - 2026-05-26`, `65.115.0 - 2026-05-19`:
    https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/CHANGELOG.md
  - Python `2.176.0 - 2026-05-26`, `2.175.0 - 2026-05-19`:
    https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/CHANGELOG.rst
  - Java `3.88.0 - 2026-05-26`, `3.87.0 - 2026-05-19`:
    https://github.com/oracle/oci-java-sdk/blob/013bb11dca68f84ec10a8e48e99855dfbb3a8f13/CHANGELOG.md
  - TypeScript `2.133.0 - 2026-05-26`, `2.132.0 - 2026-05-19`:
    https://github.com/oracle/oci-typescript-sdk/blob/8c11d0c62800f284cc12d59cb9052518d1f9bea1/CHANGELOG.md
  - The 2026-05-26 entries are word-for-word the same across Go and Python
    (e.g. "Support for ... me-alrayyan-1 region", the GoldenGate ZeroETL bullets,
    the Data Science IDCS breaking change). **[INFERENCE]** Same spec batch →
    one generation run → simultaneous multi-language publish. This is the single
    strongest indicator of a centralized internal pipeline.

- **?? verify:** The actual CI/build orchestration (internal Jenkins/Wercker?) is
  not public. `oci-python-sdk` retains a `wercker.yml`
  (https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/wercker.yml)
  — a legacy hint that Wercker (an Oracle-acquired CI) was/is used for test/build,
  but the generation itself runs in an internal pipeline upstream of GitHub.

## 4. Per-language customization mechanisms (handwritten vs generated mixing)

**Finding: The pattern is uniform — generated service clients + models + waiters,
sitting on top of a hand-written `common`/`auth` core (signing, retries, region
logic), plus a small set of hand-written service "extensions" (notably Object
Storage upload/download transfer managers).**

- **Hand-written core, by language:**
  - Go: `common/` (client plumbing, signing, retry, regions) and `common/auth/`
    are listed as `NON_GEN_TARGETS`. Signing file has **no** generation header:
    https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/common/http_signer.go#L1-L18
    Auth providers (instance principals, OKE workload identity, federation) are
    hand-written under `common/auth/`:
    https://github.com/oracle/oci-go-sdk/tree/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/common/auth
  - Python: `src/oci/signer.py` (no generator header — hand-written) and
    `src/oci/auth/`, `src/oci/base_client.py`:
    https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/signer.py#L1-L20
  - Java: the entire `bmc-common` module is hand-written; signing lives at
    `com/oracle/bmc/http/signing/` (interfaces + `internal/RequestSignerImpl`):
    https://github.com/oracle/oci-java-sdk/blob/013bb11dca68f84ec10a8e48e99855dfbb3a8f13/bmc-common/src/main/java/com/oracle/bmc/http/signing/internal/RequestSignerImpl.java#L1-L8
    Java also factors HTTP client choice into separate hand-written modules
    (`bmc-common-httpclient`, `bmc-common-httpclient-choices`).
  - TypeScript: `lib/common/lib/signer.ts` (copyright header only, no
    auto-generated marker — hand-written) and `lib/common/lib/auth/`:
    https://github.com/oracle/oci-typescript-sdk/blob/8c11d0c62800f284cc12d59cb9052518d1f9bea1/lib/common/lib/signer.ts#L1-L16

- **Generated "convenience" layers are still generated:** waiters /
  composite-operations are produced by the generator, e.g. Python
  `object_storage_client_composite_operations.py` carries the
  `OracleSDKGenerator ... API Version: 20160918` header:
  https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/object_storage/object_storage_client_composite_operations.py#L1-L6

- **Hand-written service extensions (the canonical exception):** the Object
  Storage **transfer/upload manager** is hand-authored in every SDK:
  - Go: `objectstorage/transfer/` (a `NON_GEN_TARGET`, excluded from clean):
    https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/Makefile#L4
  - Python: `src/oci/object_storage/transfer/upload_manager.py` has **no**
    generator header:
    https://github.com/oracle/oci-python-sdk/blob/b84a275083f27c6d56b635a7ad60f477d1a2849b/src/oci/object_storage/transfer/upload_manager.py
  - Java: separate Maven sub-module `bmc-objectstorage-extensions` (vs the
    `bmc-objectstorage-generated` sub-module). The generated/extensions split is
    Java's explicit customization mechanism:
    https://github.com/oracle/oci-java-sdk/tree/013bb11dca68f84ec10a8e48e99855dfbb3a8f13/bmc-objectstorage

- **[INFERENCE] Customization expression:** Unlike AWS Smithy customizations or
  AutoRest directives (which are public), OCI's customization config is internal.
  Publicly, the only visible "customization seam" is the **module/directory
  split** (generated dir/module vs hand-written `extensions`/`transfer`/`common`),
  enforced by build files like the Go Makefile's `GEN_TARGETS` vs
  `NON_GEN_TARGETS`. We cannot see per-service generator overrides directly.
  **?? verify:** whether OCI uses per-service customization config files
  (analogous to Smithy customizations) — not visible in public repos.

service, confirming per-service specs feeding one generator.

## 6. Release & versioning

- **[EVIDENCE] Per-language independent semantic version, NOT shared:**
  latest tags at research time — Go `v65.116.0`, Python `v2.176.0`,
  Java `v3.88.0`, TypeScript `v2.133.0`. (gh `repos/oracle/<repo>/tags`.)
  Go's major `65` is a Go-modules artifact (`/v65` import path), seen in imports
  like `github.com/oracle/oci-go-sdk/v65/common`:
  https://github.com/oracle/oci-go-sdk/blob/ae06daeddf367a7e56c35eb5a8b9bf2ceb8f9368/objectstorage/objectstorage_client.go#L13-L17
- **[EVIDENCE] Synchronized release *dates*, whole-SDK (monorepo) versioning,
  not per-service.** Each release is a single version bump for the entire
  monorepo (all services), published the same week across languages (see the
  §3 CHANGELOG evidence). There is no per-service version number.
- **[INFERENCE]** Cadence is roughly weekly (entries dated 2026-05-26, -05-19,
  -05-12, -05-05, -04-28 in the Java/TS changelogs). A release cuts whatever
  service spec changes accumulated that cycle, regenerates all languages, and
  bumps each language's own running version.

## Summary of evidence quality

- **Directly evidenced:** generator name `OracleSDKGenerator`; OpenAPI/Swagger
  IDL with per-service date versions; monorepo-per-language layout; generated vs
  hand-written split (Makefile `GEN_TARGETS`/`NON_GEN_TARGETS`, Java
  `-generated`/`-extensions` modules); hand-written cavage-draft signing in all
  4 languages; synchronized weekly multi-language releases; CLI generated by the
  same tool.
- **Inferred:** single shared internal spec registry feeding one generation run;
  internal CI orchestration; the existence/shape of any per-service customization
  config.
- **Open gaps (`?? verify`):** (a) downloadable raw spec endpoint on
  docs.oracle.com; (b) the internal CI/build system specifics; (c) whether
  per-service generator override/customization config exists analogous to Smithy
  customizations.
