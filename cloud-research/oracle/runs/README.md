# OCI local generator run — IMPOSSIBLE (closed-source generator)

**Result: CANNOT RUN.** Unlike the other four providers, OCI's SDK generator is
closed-source and unpublished, so there is no tool to invoke locally. This file
documents the attempt and the evidence, rather than papering over the gap.

## What was attempted
1. **Searched for any public generator artifact.** All plausible repository names
   return HTTP 404 (checked via `api.github.com/repos/...` on 2026-05-30):
   - `oracle/oci-sdk-generator` → 404
   - `oracle/OracleSDKGenerator` → 404
   - `oracle/oci-codegen` → 404
   - `oracle/oci-sdk-codegen` → 404
   - `oracle/oci-generator` → 404
   - `oracle/oci-api-spec` → 404 (no public spec repo either)
2. **Confirmed the generator exists but only as a name in generated headers.**
   The tool identifies itself as **`OracleSDKGenerator`** in every published SDK
   (see `oracle/notes.md` §2): e.g. Java
   `@jakarta.annotation.Generated(value = "OracleSDKGenerator", comments = "API
   Version: 20160918")`, Python/TS `NOTE: This class is auto generated by
   OracleSDKGenerator. DO NOT EDIT.` An authenticated org-wide code search for
   `OracleSDKGenerator org:oracle` returns ~1,900 hits — all of them *generated
   output*, never the generator's own source.
3. **No spec to feed a generator even if we had one.** OCI does not publish the
   per-service OpenAPI/Swagger spec as a raw file in any public repo; only the
   rendered API reference at `docs.oracle.com/en-us/iaas/api/` is public, and it
   is a JS SPA with no confirmed downloadable raw-spec endpoint.

## Why this is the faithful outcome (not a workaround)
Per the project rule (no silent fallbacks), the right result here is an explicit
"impossible," not a substitute. Any local "OCI run" we could fake — e.g. feeding
a reverse-engineered spec to a *different* generator — would not be OCI's pipeline
and would misrepresent it. The closed-source generator is itself a finding: OCI is
the only one of the five providers whose generator a third party cannot obtain or
run.

## Indirect evidence the pipeline is real and centralized (from the notes)
Even without running it, the pipeline's existence is strongly evidenced:
- **Synchronized weekly multi-language releases** with word-for-word-identical
  feature lists across Go / Python / Java / TypeScript on the same dates
  (`oracle/notes.md` §3, re-confirmed in `oracle/verification.md`) — one internal
  generation run, four simultaneous publishes.
- **A public client for an internal cross-language test harness,
  `SDKTestingService`** (`oracle/verification.md`), shows OCI drives every
  language SDK against one shared, internally-served test corpus — i.e. the
  generation *and* validation are centralized; only the published output is open.

## Takeaway for the deliverable
OCI is the control case that proves the comparison's "open-source posture" axis
matters: the same industrial multi-language pattern (one spec → one generator →
N synchronized SDKs, validated by a shared cross-language harness) can be run
entirely behind closed doors. We can observe its *outputs* in detail but cannot
reproduce its *process*.
