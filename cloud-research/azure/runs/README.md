# Azure local generator runs — the dual stack, run both ways

Azure is mid-migration, so two runs were done to capture **both** codegen stacks
on the same toy service:

| Subdir | Path | Spec format | Generator | Target |
|---|---|---|---|---|
| `widget-autorest-python/` | **legacy** | Swagger 2.0 (`widgets.json`) | AutoRest `@autorest/python@6.50.3` | Python |
| `widget-typespec-ts/` | **new** | TypeSpec (`main.tsp`) | `@azure-tools/typespec-ts` 1.x | TypeScript |

Both **SUCCEEDED**.

## 1. Legacy — AutoRest → Python  (`widget-autorest-python/`)
- Toolchain: Node 26 + `autorest` CLI 3.8.0. AutoRest self-downloads its core
  (`@autorest/core` 3.10.9), the `@autorest/python` 6.50.3 plugin, and
  `@autorest/modelerfour` 4.23.7 at runtime.
- **Live deprecation evidence:** AutoRest printed
  `⚠️ AutoRest is deprecated and will be retired on July 1, 2026. Please migrate
  to TypeSpec` — the same retirement the deliverable cites from the repo readme,
  observed first-hand at runtime.
- Output: 18 files incl. sync + async (`aio/`) clients, `_configuration.py`,
  `_utils/serialization.py`, `operations/_operations.py`.
- **Architecture confirmations:**
  - Generation header verbatim: `# Code generated by Microsoft (R) AutoRest Code
    Generator (autorest: 3.10.9, generator: @autorest/python@6.50.3) Changes may
    cause incorrect behavior and will be lost if the code is regenerated.`
  - The **`_patch.py` escape hatch** is emitted at every level (`_patch.py`,
    `aio/_patch.py`, `operations/_patch.py`, `aio/operations/_patch.py`) — the
    documented "last-resort customization that survives regeneration" seam.
  - Generated code imports `azure.core` (the hand-written runtime).

## 2. New — TypeSpec → TypeScript  (`widget-typespec-ts/`)
- Toolchain: `@typespec/compiler` 1.12.0 + `@typespec/http`/`@typespec/rest`
  + `@azure-tools/typespec-ts` + `@azure-tools/typespec-client-generator-core`.
- Input: `main.tsp` (a `@service` namespace, a `Widget` model, a `Widgets`
  interface) + `tspconfig.yaml` selecting `emit: ["@azure-tools/typespec-ts"]`.
  Note `main.tsp` imports `Azure.ClientGenerator.Core` — the same library whose
  `client.tsp` decorators the deliverable cites as Azure's customization seam.
- Invocation: `npx tsp compile main.tsp`.
- Output: a full **modular** TS SDK — `src/widgetServiceClient.ts`,
  `src/models/{index,models}.ts` (interface + `widgetDeserializer`),
  `src/api/widgets/operations.ts`, a `classic/` convenience layer, plus
  `package.json`, `tsconfig.json`, `api-extractor.json`, `rollup.config.js`.
- **Architecture confirmation:** generated client binds to the runtime
  `@typespec/ts-http-runtime` (`import { Pipeline } from
  "@typespec/ts-http-runtime"`), i.e. generated surface over a hand-written
  pipeline — the same generated/hand-written split as the AutoRest path.

## Takeaway for the deliverable
The two runs make Azure's "dual stack" concrete: the *same* trivial API,
expressed as Swagger and as TypeSpec, drives two different generators into two
languages, both producing a thin generated surface over a hand-written runtime
(`azure.core` / `@typespec/ts-http-runtime`). The AutoRest run even emits its own
retirement notice — the migration is live.
