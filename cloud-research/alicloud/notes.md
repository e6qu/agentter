## TL;DR architecture

Alibaba Cloud's modern ("V2.0") multi-language SDKs are generated by **Darabonba**
(formerly "TeaDSL"; file extensions `.tea`/`.dara`). The flow is:

```
Per-product API metadata (API portal / aliyun-openapi-meta)
        │  (authored/curated into a Darabonba module)
        ▼
<service>-<version>/main.tea  +  Teafile  (in aliyun/alibabacloud-sdk monorepo)
        │  dara codegen --lang <java|go|py|ts|...>   (@darabonba/cli dispatches to
        │                                             @darabonba/<lang>-generator)
        ▼
Per-language generated SDK package (one per service-version)
   ├── Java   → aliyun/alibabacloud-java-sdk        (monorepo)
   ├── Python → aliyun/alibabacloud-python-sdk      (monorepo)
   ├── TS     → aliyun/alibabacloud-typescript-sdk  (monorepo)
   └── Go     → alibabacloud-go/<service>-<version> (SEPARATE org, one repo each)
        │  generated code calls the hand-written runtime:
        ▼
Hand-written runtime: Tea/Darabonba core (tea-python = `darabonba-core`, tea-java,
   tea-typescript, tea-go) + Tea OpenApi client (darabonba-openapi, itself .dara) +
   credentials-* (alibabacloud-credentials) + Util libs.
```

The **OSS data-plane** (`aliyun-oss-python-sdk`, `aliyun-oss-go-sdk`) is a
**bespoke hand-written** SDK that predates Darabonba — confirmed divergence
(see Q5). A separate, newer `*-sdk-v2` family (e.g. `alibabacloud-oss-go-sdk-v2`,
`alibabacloud-oss-python-sdk-v2`) is also hand-written and is NOT Darabonba output.

## Q1. Service description / IDL

**Darabonba (`.dara` / `.tea` DSL).** `aliyun/darabonba` describes itself (Chinese)
as "Darabonba 是一种用于 OpenAPI 的 DSL 语言，可以用来生成多语言的 SDK、Code Sample、Test Case 等代码"
— a DSL for OpenAPI used to generate multi-language SDKs, code samples, and test
cases. (repo description; README confirms it is "a domain-specific language for
OpenAPI applications".)

The DSL is a **real, imperative programming language**, not just declarative schema.
A service module (`main.tea`) contains:
- `model` declarations (request/response/data shapes with field annotations like
  `name=`, `description=`, `example=`),
- `function`/`api` blocks with control flow,
- `import`, `extends`, `init(...)`, `super(...)` — OO-style module composition.

Example (FC, `fc-20230330/main.tea`): imports `Util/OpenApi/OpenApiUtil/EndpointUtil`,
`extends OpenApi`, defines `getEndpoint(...)` with `if`/`return` logic, then hundreds
of `model X { ... }` declarations.
Source: `aliyun/alibabacloud-sdk` `/blob/49ec5c1.../fc-20230330/main.tea`.

**Where the `.tea` specs live.** The canonical monorepo of Darabonba module sources
for *all* products is **`aliyun/alibabacloud-sdk`** (GitHub reports its primary
language as "Tea"). At the captured SHA it has **804 top-level entries**, one
directory per `<service>-<apiversion>` (e.g. `fc-20230330`, `oss-20190517`,
`mns-open-20220119`, `rocketmq-20220801`, `ons-20190214`). Each service dir contains:
- `Teafile` — build/packaging descriptor (JSON): scope, name, version, per-language
  package coordinates + the `releases` map, `libraries` (runtime deps), `productId`,
  `apiVersions`.
- `main.tea` — the Darabonba module (the IDL+logic).
- `api-info.json` — per-API documentation/metadata index (lists `hasDoc` API names,
  `product`, `version`).
- Sometimes `async/`, `v2/` subdirs (FC has both).
Source: `aliyun/alibabacloud-sdk` `/blob/49ec5c1.../fc-20230330/` (Teafile, main.tea, api-info.json).

**Relationship between Darabonba modules and per-product API meta.** The Darabonba
module is the *human-consumable, curated* layer. The underlying per-product API
metadata (parameters, types, endpoints, error codes) originates from Alibaba's
internal API "meta" / API portal (the `api-info.json` is a generated index over that
product's API doc). The `Teafile`'s `productId`/`apiVersions` and the
per-service directory name (`fc-20230330` == product FC, API version 2023-03-30) tie
the module back to a specific product+version in the API catalog.
Source: `fc-20230330/api-info.json` (`"product":"FC","version":"2023-03-30"`),
`fc-20230330/Teafile` (`"productId":"FC","apiVersions":"2023-03-30"`).

**Older / adjacent meta repos.**
- `aliyun/aliyun-openapi-meta` ("Aliyun OpenAPI Meta Data"). Its README explicitly
  scopes it: *"阿里云OpenAPI元数据，目前仅用于 aliyun-cli 工具的构建 ... the structure is not
  stable will change further"* — i.e. this public meta repo is primarily for building
  the **aliyun-cli**, not the modern SDK generation pipeline. Layout: `metadatas/<product>/`,
  `en-US/`, `zh-CN/`, `meta.go`.
  Source: `aliyun/aliyun-openapi-meta` `/blob/2563691.../README.md` and `/tree/2563691.../metadatas`.
- The V1.0 SDK family (`aliyun-openapi-java-sdk`, `aliyun-openapi-python-sdk`, etc.)
  is the legacy generation lineage being deprecated ("will soon enter Basic Security
  maintenance"). Not Darabonba.

`?? verify`: the exact public location of the authoritative machine-readable per-API
meta that feeds Darabonba module authoring (likely the Alibaba Cloud **OpenAPI
portal / next.api.aliyun.com**, which exposes per-API metadata and "API Explorer").
The `aliyun-openapi-meta` repo is explicitly NOT that source. Add to DO_NEXT.

## Q2. Generator tool(s)

**Toolchain entrypoint: `@darabonba/cli`** (`aliyun/darabonba-cli`).
- `package.json`: name `@darabonba/cli`, version `1.2.0`, `bin: { dara: ./bin/dara.js }`,
  **license Apache-2.0**. Depends on `@darabonba/parser`, `@darabonba/annotation-parser`,
  and *all* the language generators: `@darabonba/cpp-generator`, `csharp-generator`,
  `go-generator`, `java-async-generator`, `java-generator`, `php-generator`,
  `python-generator`, `swift-generator`, `typescript-generator`, plus `@darabonba/repo-client`
  and `@alicloud/tea-fileform`.
  Source: `aliyun/darabonba-cli` `/blob/f08154d.../package.json`.
- CLI command groups (from `bin/dara.js`): `init`; `check`, `codegen`, `build`,
  `format`, `config`; `login`, `info`, `pack`, `publish`, `unpublish`, `install`,
  `clean`, `score`; `help`.
  Source: `aliyun/darabonba-cli` `/blob/f08154d.../bin/dara.js`.

**Language generators (each its own repo, all under `aliyun/`):**
- `darabonba-go-generator` — `@darabonba/go-generator`, version `2.1.4`, **Apache-2.0**,
  written in **JavaScript**. Source: `/blob/618f9f6.../package.json`.
- `darabonba-python-generator` — `@darabonba/python-generator`, primary language **Python**
  (repo). SHA `744354d`.
- `darabonba-java-generator` — `@darabonba/java-generator` (JavaScript).
- `darabonba-java-async-generator` — `@darabonba/java-async-generator` (JavaScript).
- `darabonba-typescript-generator` — `@darabonba/typescript-generator` (JavaScript).
- Plus `darabonba-csharp-generator` (C#), `darabonba-php-generator`, `darabonba-cpp-generator`,
  `darabonba-swift-generator` (referenced by CLI deps).
Most generators are themselves written in **JavaScript/Node** (they consume the
`@darabonba/parser` AST); the Python generator repo's primary language is Python.

**Generator dispatch (the actual map).** `commands/codegen.js` in the CLI holds:
```js
const generatorMap = {
  cs: require('@darabonba/csharp-generator'),
  ts: require('@darabonba/typescript-generator'),
  go: require('@darabonba/go-generator'),
  java: require('@darabonba/java-generator'),
  'java-async': require('@darabonba/java-async-generator'),
  php: require('@darabonba/php-generator'),
  py: require('@darabonba/python-generator'),
  py2: require('@darabonba/python-generator'),
  cpp: require('@darabonba/cpp-generator'),
  swift: require('@darabonba/swift-generator'),
};
const Generator = generatorMap[options.lang];
```
i.e. `dara codegen --lang <go|java|py|ts|...>` parses the `.tea` into an AST then
runs the language generator, merging per-language options from the Teafile
(`...options.pkg[generatorNameMap[options.lang]]`).
Source: `aliyun/darabonba-cli` `/blob/f08154d.../commands/codegen.js`.

**Tea runtime libraries (hand-written, separate from generators):**
- `aliyun/tea-python` — Python runtime, published as **`darabonba-core`** (setup.py:
  `PACKAGE = "darabonba"`, `setup(name='darabonba-core', ...)`). Contains
  `darabonba/core.py`, `runtime.py`, `request.py`, `response.py`, `model.py`,
  `policy/`, `utils/` — hand-written (uses `aiohttp`, `ssl`, `certifi`).
  Source: `aliyun/tea-python` `/blob/926e34f.../setup.py`, `/tree/926e34f.../darabonba`.
- `aliyun/tea-java` ("Tea core for Java", Apache-2.0), `aliyun/tea-typescript`,
  `tea-go` (Go runtime lives at `github.com/alibabacloud-go/tea`), `tea-php`, `tea-cpp`,
  `tea-csharp`, plus util libs `tea-util`, `tea-xml`.
- Tea **OpenApi** client: `aliyun/darabonba-openapi` — module name `OpenApiV2`,
  itself authored in Darabonba (`main.tea` + `utils.dara` + `Teafile`) and generated
  per language; published as `alibabacloud_tea_openapi` (py), `com.aliyun:tea-openapi`
  (java), `github.com/alibabacloud-go/darabonba-openapi/v2/client` (go),
  `@alicloud/openapi-core` (ts). Its `libraries` are `GatewaySPI` + `Credential`.
  Source: `aliyun/darabonba-openapi` `/blob/aca043e.../Teafile` (`releases`/`libraries`).

All Apache-2.0 / open source where licensed; Tea runtime + credentials are public on GitHub.

## Q3. Pipeline / orchestration

**Monorepo of sources, mixed publishing model.**
- **Sources**: `aliyun/alibabacloud-sdk` is the single monorepo of `.tea` module
  sources for all ~800 services (GitHub primary language "Tea"). One dir per
  `<service>-<version>`, each with `Teafile` + `main.tea` (+ `api-info.json`).
- **Generated output, per language**:
  - Java → `aliyun/alibabacloud-java-sdk` (monorepo, primary language Java).
  - Python → `aliyun/alibabacloud-python-sdk` (monorepo): each service is a top-level
    dir `fc-20230330/` containing a standard pip package layout (`setup.py`,
    `setup.cfg`, `MANIFEST.in`, `ChangeLog.md`, `README*.md`, and the package
    `alibabacloud_fc20230330/` with `client.py` + `models/`).
    Source: `aliyun/alibabacloud-python-sdk` `/tree/9dba772.../fc-20230330`.
  - TypeScript → `aliyun/alibabacloud-typescript-sdk` (monorepo).
  - **Go → SEPARATE GitHub org `alibabacloud-go`, one repo per service-version**
    (e.g. `alibabacloud-go/fc-20230330`, layout: `client/<...>.go`, `go.mod`,
    `ChangeLog.txt`, `README*.md`). This is the notable structural divergence: Go is
    NOT a monorepo; each module is its own Go module/repo to satisfy Go module
    versioning (the Teafile go coordinate carries the `/v4` major-version path).
    Source: `alibabacloud-go/fc-20230330` `/tree/4c39508.../client`.

**How a `.tea` module becomes packages.** For a given service dir, the `dara` CLI is
run per target language (`dara codegen --lang go|java|py|ts ...`). The Teafile supplies
the per-language package metadata (Java Maven coords + packageInfo, Python pip name +
author, Go module path, TS npm name, etc.) and the `releases` map of published versions.
The generator emits idiomatic source for that language into the corresponding output repo.
Source: `fc-20230330/Teafile` (the `java`/`python`/`go`/`ts`/`releases` blocks).

**Proof the per-language code is generated, not hand-edited:**
- Python: `fc-20230330/alibabacloud_fc20230330/client.py` begins
  `# This file is auto-generated, don't edit it. Thanks.`
  Source: `aliyun/alibabacloud-python-sdk` `/blob/9dba772.../fc-20230330/alibabacloud_fc20230330/client.py`.
- Go: `alibabacloud-go/fc-20230330/client/client.go` begins
  `// This file is auto-generated, don't edit it. Thanks.`
  Source: `alibabacloud-go/fc-20230330` `/blob/4c39508.../client/client.go`.

**CI/orchestration** `?? verify`: the exact CI that regenerates and publishes (GitHub
Actions vs internal Alibaba CI). The Teafile `releases` map is the source of truth for
published versions, suggesting a release script reads it. The `dara publish`/`pack`/
`score` CLI commands exist for module publishing to the Darabonba module repository
(`@darabonba/repo-client`). Add to DO_NEXT: find the publish workflow.

## Q4. Per-language customization mechanisms

Because Darabonba is a **real imperative DSL**, "customization" is primarily **writing
logic directly in the `.tea` module** rather than maintaining out-of-band templates/
override files (contrast: AWS Smithy customizations, AutoRest directives). Any special
endpoint logic, request shaping, pagination, or product-specific behavior is expressed
as Darabonba `function`s/`api` blocks that the generators faithfully translate to each
language. Example: FC's `getEndpoint(...)` endpoint-resolution function is hand-written
Darabonba logic in `main.tea`, compiled into every language.
Source: `fc-20230330/main.tea` (`getEndpoint`).

**Per-language knobs live in the Teafile** (not template overrides), e.g.:
- `java.enableMinimizeModelName` (true for OSS, false for FC),
- `go.mapAndSliceWithoutOmitempty: true` (FC),
- `java.package`, `python.package`, `csharp.namespace`, `php.package`, namespace/
  client-name per language, packageInfo (descriptions, license, dev metadata).
Source: `fc-20230330/Teafile`, `oss-20190517/Teafile`.

**Hand-written code coexists via the runtime layer.** Generated clients `extends
OpenApi` and import `Util/OpenApiUtil/EndpointUtil/GatewaySPI(SPI)/GatewayPOP/
Paginator` (declared in the Teafile `libraries`). The HTTP transport, retry, signing,
serialization, and auth are **not generated per service** — they live in hand-written
runtime libs:
- Tea/Darabonba core runtime (`tea-python`=`darabonba-core`, `tea-java`, `tea-typescript`,
  `alibabacloud-go/tea`) — HTTP, retry, models base.
- Tea OpenApi client (`darabonba-openapi` → `alibabacloud_tea_openapi`, `tea-openapi`,
  etc.) — the POP gateway request/response pipeline, RPC + ROA styles.
- `credentials-*` (`alibabacloud-credentials`) — credential providers/chain.
The generated Python FC client imports exactly these:
`from alibabacloud_tea_openapi.client import Client as OpenApiClient`,
`from darabonba.core import DaraCore`, `from darabonba.runtime import RuntimeOptions`.
Source: `aliyun/alibabacloud-python-sdk` `/blob/9dba772.../fc-20230330/alibabacloud_fc20230330/client.py` (imports).

Go equivalent imports: `github.com/alibabacloud-go/darabonba-openapi/v2/client`,
`.../v2/utils`, `github.com/alibabacloud-go/tea/dara`.
Source: `alibabacloud-go/fc-20230330` `/blob/4c39508.../client/client.go` (imports).

**Gateway SPI pattern.** Newer services route through a pluggable gateway (`GatewaySPI`
+ `GatewayPOP`/`GatewayClient` libraries in the Teafile), letting one generated client
support multiple backend protocols without per-service hand-code.
Source: `fc-20230330/Teafile` `libraries` (`SPI: alibabacloud:GatewaySPI:*`,
`GatewayClient: alibabacloud:GatewayPOP:*`).

## Q5. Manual vs generated (case studies)

### OSS (Object Storage Service) — the important divergence
Two distinct worlds:
1. **Control-plane / management API** IS Darabonba-generated: there is an
   `oss-20190517/` module (`Teafile` name `Oss20190517` v1.0.14, `main.tea`,
   `api-info.json`) in the monorepo, producing `alibabacloud_oss20190517` (py),
   `com.aliyun.oss20190517` (java), etc. `java.enableMinimizeModelName: true`.
   Source: `aliyun/alibabacloud-sdk` `/blob/49ec5c1.../oss-20190517/Teafile`.
2. **Data-plane object SDK is BESPOKE HAND-WRITTEN**, predating Darabonba:
   - `aliyun/aliyun-oss-python-sdk` — package **`oss2`**, **MIT** license, hand-written
     modules: `api.py`, `auth.py`, `crypto.py`, `crypto_bucket.py`, `resumable.py`,
     `xml_utils.py`, `select_response.py`, etc. `oss2/api.py` has a Chinese hand-written
     docstring, NO "auto-generated" header — confirming hand-authored.
     Source: `aliyun/aliyun-oss-python-sdk` `/blob/eb329ec.../oss2/api.py`.
   - `aliyun/aliyun-oss-go-sdk` — **MIT**, hand-written `oss/` package
     (`bucket.go`, `auth.go`, `conn.go`, `crc.go`, `client.go`, with `*_test.go`
     alongside — clearly hand-maintained). Source: `aliyun/aliyun-oss-go-sdk`
     `/tree/b6e0a2a.../oss`.
   - There is ALSO a newer hand-written **v2** line (`alibabacloud-oss-go-sdk-v2`,
     `alibabacloud-oss-python-sdk-v2`, `-java-sdk-v2`, `-csharp-sdk-v2`, `-cpp-sdk-v2`,
     `-rust-sdk-v2`) — still NOT Darabonba output; OSS keeps a dedicated SDK family
     because of streaming/multipart/crypto/resumable upload semantics that don't fit
     the generic POP/Darabonba model.
   **Conclusion: VERIFIED — OSS data-plane is a hand-written exception to the
   Darabonba mainline.**

## Q6. Release & versioning

- **Per-product, per-API-version packaging.** Package identity is `<product><apiversion>`
  (e.g. `alibabacloud_fc20230330`, `com.aliyun:fc20230330`, `@alicloud/fc20230330`,
  `github.com/alibabacloud-go/fc-20230330`). A new API version → a new package
  (`fc-open-20200310` vs `fc-open-20210406` vs `fc-20230330` coexist).
  Source: `aliyun/alibabacloud-sdk` root listing (`fc-20230330`, `fc-open-20200310`,
  `fc-open-20210406`).
- **Two version numbers per module.** The Teafile carries:
  - a `version` for the Darabonba *module* itself (e.g. FC `1.0.47`), and
  - a `releases` map of the *published package* version per language, which can differ
    across languages (FC: java/go/ts/php/python all `4.7.5`, csharp `4.6.12`, python2
    `4.1.1`). So per-language published versions can drift.
