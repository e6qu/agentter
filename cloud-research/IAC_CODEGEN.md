# How infrastructure-as-code tools generate their bindings

A source-cited comparison of how the **IaC layer** is code-generated: the
**CDK family** (AWS CDK, CDKTF, cdk8s via **jsii**), **Pulumi**, **Bicep**,
**Terraform** itself, and the **per-cloud Terraform providers** (AWS/`awscc`,
Google **Magic Modules**, Azure **pandora**/`azapi`).

Companion to [`CODE_GENERATION.md`](CODE_GENERATION.md) (the API-client SDK
layer). Same rules: every non-trivial claim carries a commit-pinned permalink;
SHAs pinned 2026-05-30; uncertain items marked `(unverified)`. Raw research:
`iac/*.md`. Local reproductions: [`iac/runs/`](iac/runs/README.md).

---

## The big idea: IaC tools are a *second* generation layer

The API-client SDKs in `CODE_GENERATION.md` are generated from API specs. The IaC
tools sit one layer up and form a **generation chain** â€” the SDK is often just the
middle of it:

```
API spec (OpenAPI / Smithy / proto / CloudFormation / ARM)
   â†’ API-client SDK            (generated â€” CODE_GENERATION.md)
   â†’ Terraform provider        (hand-written OR generated from the API/CFN spec)
   â†’ provider schema (gRPC)    (machine-readable resource model)
   â†’ CDKTF / Pulumi bindings   (generated per-language from that schema)
```

Two consequences run through everything below:

1. **The same root spec feeds multiple layers.** The CloudFormation resource
   schemas generate *both* AWS CDK's L1 constructs *and* the `terraform-provider-awscc`
   provider. `azure-rest-api-specs` generates the Azure SDKs (AutoRest/TypeSpec),
   *and* Bicep's type system (`bicep-types-az`), *and* HashiCorp's `go-azure-sdk`
   (via pandora). A Terraform provider's schema, in turn, generates *both* CDKTF
   constructs *and* Pulumi's bridged SDKs.
2. **"Multi-language" is solved three different ways.** jsii **transpiles** one
   TypeScript source into proxies; Pulumi **generates** a native SDK per language
   from a schema; Bicep is a **single DSL** (multi-language only in its serialized
   type model); Terraform providers are **Go-only**, but their schema is the input
   to the other two strategies.

## jsii and the CDK family

**jsii** ([`aws/jsii`](https://github.com/aws/jsii/blob/e14cd30cbf62a271d7dfc97154e77d84071e7024/gh-pages/content/specification/1-introduction.md),
Apache-2.0) is one engine reused by AWS CDK, CDKTF, and cdk8s. The pattern is
always: **author once in a TypeScript subset â†’ the `jsii` compiler emits a
language-agnostic `.jsii` assembly (API surface only, no implementation) â†’
`jsii-pacmak` emits idiomatic packages for the five targets** (`dotnet`, `go`,
`java`, `js`, `python` â€” verified in
[`targets/index.ts`](https://github.com/aws/jsii/blob/e14cd30cbf62a271d7dfc97154e77d84071e7024/packages/jsii-pacmak/lib/targets/index.ts#L16-L37)).
The assembly schema is fixed at `jsii/0.10.0`
([`assembly.ts`](https://github.com/aws/jsii/blob/e14cd30cbf62a271d7dfc97154e77d84071e7024/packages/%40jsii/spec/src/assembly.ts#L247-L248)).

The defining trait is the **runtime kernel bridge**: non-JS bindings do not
reimplement logic â€” they are proxies that spawn a Node.js child process hosting
the original JavaScript and marshal calls over a STDIN/STDOUT protocol
([`@jsii/runtime` README](https://github.com/aws/jsii/blob/e14cd30cbf62a271d7dfc97154e77d84071e7024/packages/%40jsii/runtime/README.md)).
This is exactly what the local run reproduced: the generated Python `greet()` body
is `jsii.invoke(self, "greet", [])` ([`iac/runs/jsii/`](iac/runs/jsii/README.md)).
Because of this, jsii imposes a **"jsii-compatible subset"** of TypeScript â€”
behavioural `I`-prefixed interfaces vs behaviour-free `readonly` `*Props` structs,
no method overloads, no generics, restricted unions
([type-system spec](https://github.com/aws/jsii/blob/e14cd30cbf62a271d7dfc97154e77d84071e7024/gh-pages/content/specification/2-type-system.md))
â€” which is why every CDK API reads `new Bucket(scope, id, { ...props })`.

The three CDKs differ only in **where the TypeScript comes from**:

- **AWS CDK** ([`aws/aws-cdk`](https://github.com/aws/aws-cdk/tree/32009391675de97b2394435b687f33b97b914cc3)).
  L1 (`Cfn*`) constructs are **generated** from CloudFormation resource schemas
  (normalized in
  [`cdklabs/awscdk-service-spec`](https://github.com/cdklabs/awscdk-service-spec/blob/a99a3d0c98d9b2f13311f09eb102d7b38bb21be3/README.md))
  by **`@aws-cdk/spec2cdk`** (the `gen-cdk` bin, an explicit `cfn2ts` replacement;
  [`generate.ts`](https://github.com/aws/aws-cdk/blob/32009391675de97b2394435b687f33b97b914cc3/tools/%40aws-cdk/spec2cdk/lib/generate.ts#L1-L11)
  renders via the `@cdklabs/typewriter` AST library â€” not string templates). L2/L3
  constructs (e.g. hand-written
  [`aws-s3/lib/bucket.ts`](https://github.com/aws/aws-cdk/tree/32009391675de97b2394435b687f33b97b914cc3/packages/aws-cdk-lib/aws-s3/lib))
  are authored by hand; the `*.generated.ts` L1s are gitignored build artifacts.
  The whole `aws-cdk-lib` is then jsii-published to all languages (per-module
  [`.jsiirc.json`](https://github.com/aws/aws-cdk/blob/32009391675de97b2394435b687f33b97b914cc3/packages/aws-cdk-lib/aws-s3/.jsiirc.json)).
- **CDKTF** ([`hashicorp/terraform-cdk`](https://github.com/hashicorp/terraform-cdk/blob/a0c56138e2b4c7b6c128bd7ba08a8e317df852af/packages/cdktf-cli/src/bin/cmds/get.ts#L13-L40),
  MPL-2.0). `cdktf get` obtains a provider schema by shelling out to
  `terraform providers schema -json`
  ([`provider-schema.ts`](https://github.com/hashicorp/terraform-cdk/blob/a0c56138e2b4c7b6c128bd7ba08a8e317df852af/packages/%40cdktf/provider-schema/src/provider-schema.ts#L176-L215)),
  generates TS constructs (`TerraformProviderGenerator`), then runs `jsii` +
  `jsii-pacmak --code-only`
  ([`constructs-maker.ts`](https://github.com/hashicorp/terraform-cdk/blob/a0c56138e2b4c7b6c128bd7ba08a8e317df852af/packages/%40cdktf/provider-generator/lib/get/constructs-maker.ts#L24-L25)).
  *(Note: CDKTF entered maintenance/deprecation in late 2025 â€” `(unverified)` exact
  date.)*
- **cdk8s** ([`cdk8s-team/cdk8s-cli`](https://github.com/cdk8s-team/cdk8s-cli/blob/76e5409f3a772268980c69774f8a9895c9cd52a7/src/cli/cmds/import.ts#L13-L21)).
  `cdk8s import` reads the Kubernetes OpenAPI `_definitions.json` or a CRD's
  `openAPIV3Schema` and runs `jsii-srcmak`
  ([`base.ts`](https://github.com/cdk8s-team/cdk8s-cli/blob/76e5409f3a772268980c69774f8a9895c9cd52a7/src/import/base.ts#L40-L135)).

## Pulumi â€” schema â†’ native SDK per language

Pulumi is the **opposite of jsii**: there is no shared source library to transpile.
A language-neutral **Pulumi Package Schema** (`schema.json`, spec'd by the
metaschema
[`pkg/codegen/schema/pulumi.json`](https://github.com/pulumi/pulumi/blob/979eed35363d9047e092a1e3c59bbfa28c7103a4/pkg/codegen/schema/pulumi.json))
is the only shared artifact, and **five independent generators â€” all written in
Go â€” each emit a fully idiomatic, native SDK**: Go/Python/Node.js in
[`pulumi/pulumi`](https://github.com/pulumi/pulumi/blob/979eed35363d9047e092a1e3c59bbfa28c7103a4/pkg/codegen/go/gen.go#L4781),
.NET in
[`pulumi-dotnet`](https://github.com/pulumi/pulumi-dotnet/blob/0ec9ac3f7b8c458e24b6dec1fefc2c64377314c3/pulumi-language-dotnet/codegen/gen.go#L2668),
Java in
[`pulumi-java`](https://github.com/pulumi/pulumi-java/blob/b4d6735000821b88134f90325e28d43a3415a5c1/pkg/codegen/java/gen.go#L2237).
Everything is addressed by a `pkg:module:Member` token. `pulumi package gen-sdk`
binds the schema and dispatches over gRPC to a per-language host (`pulumi-language-*`)
whose `GeneratePackage` RPC calls the Go codegen â€” hence the generated banner
`Code generated by pulumi-language-go`. The local run reproduced this: one
`schema.json` â†’ a real `class Greeter(pulumi.ComponentResource)` **and** a real
`func NewGreeter(...)` ([`iac/runs/pulumi/`](iac/runs/pulumi/README.md)).

**Most cloud providers are bridged from Terraform.** `pulumi-aws`, `pulumi-gcp`,
and `pulumi-azure` (classic) are not hand-authored schemas â€” the
[`pulumi-terraform-bridge`](https://github.com/pulumi/pulumi-terraform-bridge/blob/15fe56807bdd30d0c03fc1d394ecb33445694718/README.md#L33-L41)
introspects the upstream Terraform provider's schema and emits a standard Pulumi
Package Schema
([`generate_schema.go`](https://github.com/pulumi/pulumi-terraform-bridge/blob/15fe56807bdd30d0c03fc1d394ecb33445694718/pkg/tfgen/generate_schema.go#L231-L283)),
then calls the *same* `gen-sdk` path. Verified end-to-end in `pulumi-aws`: the TF
type `aws_acmpca_certificate_authority` â†’ token
`aws:acmpca/certificateAuthority:CertificateAuthority`
([`resources.go` with `GitHubOrg: "hashicorp"`](https://github.com/pulumi/pulumi-aws/blob/f0e90c4722b549b5e4a1a06438660f096f9e4e9f/provider/resources.go#L906-L935))
â†’ generated
[`acmpca.CertificateAuthority`](https://github.com/pulumi/pulumi-aws/blob/f0e90c4722b549b5e4a1a06438660f096f9e4e9f/sdk/go/aws/acmpca/certificateAuthority.go#L1-L45)
in every language. Native (non-bridged) providers emit the *same* schema by other
means: `pulumi-azure-native` from
[Azure OpenAPI](https://github.com/pulumi/pulumi-azure-native/blob/f0746b7101b9c07b4e354c8dfddf3f3ff74a2415/provider/pkg/gen/schema.go#L80-L81),
`pulumi-google-native` from
[Google Discovery docs](https://github.com/pulumi/pulumi-google-native/blob/3000d16840aa1b22936f48beb16bec6afb05a6bf/provider/pkg/gen/schema.go#L38-L45).

Customization lives in the bridge's
[`ProviderInfo`](https://github.com/pulumi/pulumi-terraform-bridge/blob/15fe56807bdd30d0c03fc1d394ecb33445694718/pkg/tfbridge/info/info.go#L48-L135)
(token renames, per-field overrides, the ubiquitous `MaxItemsOne` flattening,
`Extra*` injected schema) and, schema-wide, the `language` bag and `isOverlay`
(the marker for a hand-written shim). All Pulumi repos are Apache-2.0.

## Bicep â€” a single DSL over a generated type system

Bicep ([`Azure/bicep`](https://github.com/Azure/bicep), MIT) is a single
purpose-built DSL implemented in C#/.NET that **transpiles `.bicep` â†’ ARM JSON
deployment templates** (reproduced in [`iac/runs/bicep/`](iac/runs/bicep/README.md):
`metadata._generator.name == "bicep"`). Unlike CDK/Pulumi it has **no
multi-language authoring** â€” the only multi-language surface is the *serialization*
of its type model.

The codegen story is that type model. **`Azure/bicep-types`**
([README](https://github.com/Azure/bicep-types/blob/f1de99c672dd23d7e14f2647e38e4f41ee421dee/README.md))
defines a language-agnostic type representation (with parallel TS, C#, and Go
implementations). **`Azure/bicep-types-az`** generates the Azure resource types
**from `azure-rest-api-specs`** â€” the *same* Swagger/TypeSpec specs that drive the
Azure SDKs and AutoRest â€” via an AutoRest extension
([`src/autorest.bicep/src/main.ts`](https://github.com/Azure/bicep-types-az/blob/94edabb1b9de78aa1adb4a8f8231375a73cd2016/src/autorest.bicep/src/main.ts),
orchestrated by
[`generate.ts`](https://github.com/Azure/bicep-types-az/blob/94edabb1b9de78aa1adb4a8f8231375a73cd2016/src/generator/src/cmd/generate.ts)),
emitting per-provider/version
[`types.json`](https://github.com/Azure/bicep-types-az/blob/94edabb1b9de78aa1adb4a8f8231375a73cd2016/generated/storage/microsoft.storage/2024-01-01/types.json).
So even Bicep's authoring experience is a downstream of `azure-rest-api-specs`,
making it a sibling â€” at the IaC layer â€” of the AutoRest/TypeSpec SDK pipeline.
(The same generated types are reused by `azapi`; see below.)

## Terraform itself â€” protocol, plugin frameworks, and spec codegen

A Terraform provider is a plugin that talks to core over **gRPC**; the wire
contract (`tfplugin6.proto`) and the provider **schema** it exposes via
`GetProviderSchema` live in
[`terraform-plugin-go`](https://github.com/hashicorp/terraform-plugin-go/blob/9626331c59cfb5aa6db7e6c3e77dae9f4d0b7679/tfprotov6/schema.go).
This schema â€” projected to JSON by `terraform providers schema -json`
([core command](https://github.com/hashicorp/terraform); reproduced in
[`iac/runs/terraform-schema/`](iac/runs/terraform-schema/README.md)) â€” is the
machine-readable model that **CDKTF and the Pulumi bridge consume**.

There are two ways to *author* a provider, both hand-written Go:
- **Plugin SDK v2** ([`terraform-plugin-sdk`](https://github.com/hashicorp/terraform-plugin-sdk/blob/9e392b01d50810502107495b880e756a13878195/helper/schema/resource.go))
  â€” the classic map-based `schema.Resource` + CRUD.
- **Plugin Framework** ([`terraform-plugin-framework`](https://github.com/hashicorp/terraform-plugin-framework/blob/350c632115f1bb64318a89a5956aa0283807c4b8/resource/schema/schema.go))
  â€” the newer typed approach.

And a newer **spec-based codegen** path: a JSON **provider code specification**
([`terraform-plugin-codegen-spec`](https://github.com/hashicorp/terraform-plugin-codegen-spec/blob/f777f36547cb82a9f4ad710ae64d06f8cbca2520/spec/v0.1/schema.json)),
which [`tfplugingen-openapi`](https://github.com/hashicorp/terraform-plugin-codegen-openapi/blob/cb5cfbd18ac9e9e497e9986cbd2239d855936d01/README.md)
generates *from* an OpenAPI 3 document, and
[`tfplugingen-framework`](https://github.com/hashicorp/terraform-plugin-codegen-framework)
turns *into* Plugin Framework Go. The local run reproduced the second step
([`iac/runs/terraform-plugin-codegen/`](iac/runs/terraform-plugin-codegen/README.md)).
The crucial limit, true of every Terraform provider: **codegen produces schema +
data model only â€” CRUD bodies stay hand-written** (or are delegated to a generic
API; see `awscc`). Licenses: Terraform core is BSL 1.1; all plugin libraries and
codegen tools are MPL-2.0.

## The per-cloud Terraform providers â€” a hand-writtenâ†”generated spectrum

The three cloud providers span the full range, which is itself a finding.

> build-model matrix live in [`terraform-providers/`](terraform-providers/README.md).

### AWS â€” two providers, opposite philosophies

- **`terraform-provider-aws`** ([`e26a14e`](https://github.com/hashicorp/terraform-provider-aws/tree/e26a14e0d078679494673567ab79363a7c1a60a0/internal/generate))
  is **mostly hand-written** Go on the AWS SDK for Go v2; its `internal/generate/`
  toolbox (~30 generators driven by `//go:generate`) emits only **scaffolding** â€”
  service registration
  ([`servicepackage/main.go`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/internal/generate/servicepackage/main.go),
  which regex-scans `// @FrameworkResource` annotations on hand-written
  resources), tag helpers, paginators. Resource CRUD is authored by hand; the
  SDKv2â†’Plugin-Framework migration is happening resource by resource.
- **`terraform-provider-awscc`** ([`e9fd27e`](https://github.com/hashicorp/terraform-provider-awscc/tree/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators))
  is **fully generated** â€” 1,366 `*_resource_gen.go` files emitted from
  **CloudFormation resource schemas** (the same
  [`AWS_Logs_AccountPolicy.json`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/service/cloudformation/schemas/AWS_Logs_AccountPolicy.json)-style
  schemas AWS CDK's L1 uses) via the AWS Cloud Control API, parsed with
  `cfschema` and rendered through `codegen.Emitter`. No per-resource CRUD â€” every
  resource delegates to the generic `cloudcontrol` Create/Read/Update/Delete.
  **Same cloud, same CloudFormation source as AWS CDK â€” opposite build philosophy
  to its sibling provider.**

### Google â€” the most thoroughly generated provider (Magic Modules)

[`GoogleCloudPlatform/magic-modules`](https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1)
generates **both** `terraform-provider-google` and `-google-beta` from a single set
of YAML resource definitions. Each resource is a
[`mmv1/products/<product>/<Resource>.yaml`](https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/products/pubsub/Topic.yaml#L14-L56)
(fields, CRUD URLs, async behaviour) rendered through Go `text/template` `.tmpl`
templates
([`resource.go.tmpl`](https://github.com/GoogleCloudPlatform/magic-modules/blob/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/templates/terraform/resource.go.tmpl#L1-L21)).
The generated-file header is self-documenting: `Type: MMv1` cites the source YAML +
template, while `Type: Handwritten` cites a
[`third_party/terraform/`](https://github.com/GoogleCloudPlatform/magic-modules/tree/8cc65df4a76512da5e02abe7fcfd39657374b906/mmv1/third_party/terraform)
override (e.g. `google_storage_bucket`, whose `Bucket.yaml` sets
`exclude_resource: true`). Notable recent change: the engine's **Rubyâ†’Go migration
is complete** â€” the last Ruby remnants and the older DCL-based `tpgtools` path were
removed on 2026-05-18 in
[PR #17608](https://github.com/GoogleCloudPlatform/magic-modules/pull/17608)
(15,415 deletions). One YAML edit â†’ `make provider VERSION=ga/beta` â†’ both
providers regenerated from the same source. Generator/config is Apache-2.0; the
`third_party` Go copied into providers is MPL-2.0.

### Azure â€” hand-written over a self-generated SDK (pandora)

- **`terraform-provider-azurerm`** is **mostly hand-written** Go (SDKv2 +
  Plugin-Framework muxed), but it calls into **`hashicorp/go-azure-sdk`**.
- **`go-azure-sdk` is generated by [`hashicorp/pandora`](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/README.md)
  from `azure-rest-api-specs`** â€” i.e. HashiCorp generates its *own* Azure SDK
  rather than using Microsoft's `azure-sdk-for-go`. Pandora's pipeline:
  `importer-rest-api-specs` parses the swagger (wired as a git submodule) â†’
  `data-api` serves an intermediate model
  ([`api-definitions/`](https://github.com/hashicorp/pandora/tree/dca5fb0f724ef92619919810f0e0a216a57d2e66/api-definitions))
  â†’ [`generator-go-sdk`](https://github.com/hashicorp/pandora/tree/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-go-sdk/internal/generator)
  emits the SDK (the
  [`helpers.go` `copyrightLinesForSource`](https://github.com/hashicorp/pandora/blob/dca5fb0f724ef92619919810f0e0a216a57d2e66/tools/generator-go-sdk/internal/generator/helpers.go#L77)
  emits the generated-file header keyed on an `AzureRestAPISpecs` source origin).
- **`terraform-provider-azapi`** is **schema-driven** â€” one generic resource type,
  no per-resource Go, embedding generated Azure
  [type definitions](https://github.com/Azure/terraform-provider-azapi/tree/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/internal/azure/generated)
  copied from the **Bicep** types project
  ([`scripts/bicep-types-update.sh`](https://github.com/Azure/terraform-provider-azapi/blob/efeb3ca3c0dee7d75a1ebde92ae58a7be587715b/scripts/bicep-types-update.sh)).

So in Azure, `azure-rest-api-specs` is the common root of *four* generated
artifacts: `azure-sdk-for-*` (AutoRest/TypeSpec), `bicep-types-az` (Bicep +
azapi), and `go-azure-sdk` (pandora â†’ azurerm). All four HashiCorp repos are
MPL-2.0.

## Side-by-side: the multi-language strategy

| Strategy | Tools | How | Trade-off |
|---|---|---|---|
| **Transpile one source** | AWS CDK, CDKTF, cdk8s (jsii) | author 1 TS lib â†’ proxies that call a Node kernel | one implementation to maintain; but every app embeds a Node runtime, and the API is constrained to the jsii subset |
| **Generate per language** | Pulumi | 5 Go generators read one schema â†’ native SDKs | idiomatic, runtime-free SDKs; but 5 generators to maintain (the cloud-SDK model) |
| **Single DSL** | Bicep | one compiler â†’ ARM JSON; types generated from specs | simplest; but Azure-only and one language |
| **Go + downstream schema** | Terraform providers | hand-written/generated Go; schema feeds CDKTF/Pulumi | the provider is Go-only, but its schema becomes *others'* multi-language input |

## Concluding observations

1. **IaC codegen is a chain, not a layer.** Every IaC tool is downstream of an API
   spec, often *through* a generated SDK. The same root spec (CloudFormation,
   `azure-rest-api-specs`, a TF provider schema) routinely feeds two or three
   generation layers at once.

2. **"Multi-language" has three real answers, and they recur.** Transpile-to-proxies
   (jsii), generate-per-language (Pulumi â€” identical to the cloud SDKs), or
   single-DSL (Bicep). Terraform providers opt out (Go-only) but become *input* to
   the first two.

3. **The hand-writtenâ†”generated spectrum is widest at the Terraform-provider layer.**
   `awscc` and Magic Modules are almost fully generated; `terraform-provider-aws` and
   `azurerm` are hand-written with generated scaffolding/SDKs underneath â€” sometimes
   *for the same cloud* (`awscc` vs `aws`).

4. **HashiCorp re-generates vendors' SDKs rather than consuming them.** `go-azure-sdk`
   (pandora) is HashiCorp's own Azure SDK generated from Microsoft's specs, in
   preference to `azure-sdk-for-go` â€” a striking case of the same spec generating
   two independent SDK families.

5. **CloudFormation and `azure-rest-api-specs` are super-sources.** CloudFormation
   schemas generate AWS CDK L1 *and* `awscc`. `azure-rest-api-specs` generates the
   Azure SDKs, Bicep's types, azapi's types, *and* `go-azure-sdk`. The spec, not the
   tool, is the durable asset.
