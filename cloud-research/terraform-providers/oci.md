> Repo: **`oracle/terraform-provider-oci`** (Oracle publishes and maintains it).
> License: **MPL-2.0** (`Mozilla Public License Version 2.0`).
> Pinned commit for all permalinks below: `5eab52881903a807d3875c92e314ab085b13c848`
> (HEAD of `master`, 2026-05-30).
>
> **Evidence basis.** As with the OCI SDKs, OCI's code generator is closed-source,
> so the provider's *generation* cannot be observed directly. Claims are tagged
> **[EVIDENCE]** (verified from a file/commit) or **[INFERENCE]** (reasoned from
> structure). `?? verify:` marks open gaps. The published repo contains **no**
> generator, templates, or generation headers — the public artifact is the
> generated *output*, hand-edited and reviewed like a normal Go repo.

---

## TL;DR

- The provider sits on top of **`github.com/oracle/oci-go-sdk/v65`** (the generated
  Go SDK). Chain: internal OpenAPI/Swagger spec → `oci-go-sdk` (generated by
  `OracleSDKGenerator`) → terraform-provider-oci → Terraform schema. **[EVIDENCE]**
- It uses **Plugin SDK v2 for ~all resources/data sources**, with a **tiny**
  Plugin Framework footprint (one data source), the two muxed together with
  `terraform-plugin-mux`. **[EVIDENCE]**
- The per-service resource/data-source files are **NOT marked as generated** —
  there is **no `// Code generated. DO NOT EDIT.` and no `OracleSDKGenerator`
  header anywhere in the repo**. **[EVIDENCE]** Whether they are nonetheless
  produced by an internal generator is **[INFERENCE]** (strong structural
  signals: rigid naming/CRUD patterns, per-service `*_export.go` "hints" with
  explicit "Custom overrides" hooks, weekly lockstep with the SDK). The published
  files are the editable source of truth in this repo regardless. `?? verify:`
  existence/shape of an internal provider generator.
- **Release cadence is weekly and tracks the SDK**, but on its **own version
  line** (`8.16.0`, not the SDK's `v65.x`), typically one day after the SDK cut.
  **[EVIDENCE]**

## 1. Generated or hand-written? (the key question)

**Finding: the published per-service files carry NO generation marker. There is
no shared `OracleSDKGenerator` header on the Terraform provider, unlike the SDKs.
The provider is, in the public repo, treated as hand-editable Go code. Internal
generation is plausible but unproven from public sources.**

- **[EVIDENCE] No `OracleSDKGenerator` and no `// Code generated. DO NOT EDIT.`
  anywhere.** Org-style code searches against the repo return zero hits:
  - `repo:oracle/terraform-provider-oci "Code generated" "DO NOT EDIT"` → **0**
  - `repo:oracle/terraform-provider-oci OracleSDKGenerator` → **0**
  The only `"auto generated"` hits (6 total) are incidental: an example `.tf`
  file, three doc `.html.markdown` files, and two in-code comments — e.g. a test
  comment in
  [`internal/service/queue/queue_queue_resource.go` L687](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/queue/queue_queue_resource.go#L687)
  ("This is an auto generated code for channelIds…"). None is a file header.

- **[EVIDENCE] The headers that DO exist are just copyright/license**, identical
  across resource files, with no tool name or DO-NOT-EDIT clause. Example, the
  Object Storage bucket resource:
  ```go
  // Copyright (c) 2017, 2024, Oracle and/or its affiliates. All rights reserved.
  // Licensed under the Mozilla Public License v2.0
  package objectstorage
  ```
  [`internal/service/objectstorage/objectstorage_bucket_resource.go` L1-L24](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/objectstorage_bucket_resource.go#L1-L24).
  Compare the SDK, which stamps `// Code generated. DO NOT EDIT.` (Go) and
  `auto generated by OracleSDKGenerator` (Python/TS/Java) on every client. The
  provider does **not** share that visible pipeline marker.

- **[EVIDENCE] No generator/templates checked into the repo.** The `scripts/`
  directory holds only lint/test/changelog helpers (`gofmtcheck.sh`,
  `errcheck.sh`, `docscheck.sh`, `changelog-links.sh`, pre-push hooks) —
  [`scripts/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/scripts).
  There is no `.tmpl` generator template (the single `.tmpl` in the repo is an
  example payload,
  `examples/artifacts/ContainerImageSignature/...message_json.tmpl`).

- **[EVIDENCE] CONTRIBUTING describes an ordinary human PR flow** — fork, OCA
  sign-off, "We will assign the pull request to 2-3 people for review before it
  is merged" —
  [`CONTRIBUTING.md`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/CONTRIBUTING.md).
  This is consistent with the files being directly editable, not regenerated and
  overwritten on contributors.

- **[INFERENCE — likely internally generated/scaffolded] Strong structural
  signals point to an internal generator producing skeletons that are then
  hand-finished:**
  1. **Rigid, uniform per-service layout.** 143 service packages under
     [`internal/service/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/service),
     each with the same file set: `<svc>_<resource>_resource.go`,
     `<svc>_<resource>_data_source.go`, `<svc>_<resources>_data_source.go`
     (plural list DS), `register_resource.go`, `register_datasource.go`,
     `<svc>_export.go`, and optional `helpers_<svc>.go`.
  2. **Mechanical 1:1 mapping onto the generated SDK.** The bucket resource maps
     each CRUD verb straight onto `oci-go-sdk` request/response types:
     `Create: createObjectStorageBucket` → `oci_object_storage.CreateBucketRequest{}`
     → `s.Client.CreateBucket(...)`; likewise `GetBucket`/`UpdateBucket`/
     `DeleteBucket`. See
     [`objectstorage_bucket_resource.go` L27-L36, L273, L338, L489, L519](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/objectstorage_bucket_resource.go#L27-L36).
     This boilerplate-per-field/per-verb shape is exactly what a spec-driven
     generator emits.
  3. **`*_export.go` "hints" with explicit override hooks.** Each service
     registers resource-discovery metadata and labels the customizations
     "Custom overrides…": `exportObjectStorageNamespaceHints.ProcessDiscoveredResourcesFn = ...`,
     `// Custom overrides for generating composite IDs within the resource
     discovery framework`. See
     [`objectstorage_export.go` L11-L26](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/objectstorage_export.go#L11-L26).
     A generated-skeleton + named-override-hook pattern mirrors how the SDKs split
     generated clients from hand-written extensions.
  4. **Lockstep weekly content with the SDK** (see §5) — the provider's CHANGELOG
     bullets are the SDK's spec-batch feature list, one cut behind.

  **Net:** these signals strongly suggest an internal generator scaffolds the
  per-service files (most consistent with Oracle's overall SDK pipeline), but —
  unlike the SDKs — the provider files carry **no** generation header and are
  edited/reviewed as normal source. **`?? verify:`** whether a private
  provider-generator exists and, if so, whether it shares `OracleSDKGenerator`
  infrastructure. Not determinable from public artifacts.

## 2. SDK it sits on

**Finding: built directly on the generated `oci-go-sdk/v65`.** **[EVIDENCE]**

- `go.mod` requires the SDK:
  `github.com/oracle/oci-go-sdk/v65 v65.116.0` —
  [`go.mod`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/go.mod).
  (Note `v65.116.0` is the exact SDK version pinned in the OCI SDK research at
  HEAD 2026-05-30 — the provider tracks the current SDK release.)
- Per-service client wiring lives in
  [`internal/client/*_clients.go`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/client)
  (one file per service). Each registers an SDK client factory; e.g.
  `oci_object_storage.NewObjectStorageClientWithConfigurationProvider(configProvider)`
  in
  [`object_storage_clients.go` L17-L41](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/client/object_storage_clients.go#L17-L41).
- Resource files import the generated SDK packages directly, e.g.
  `oci_object_storage "github.com/oracle/oci-go-sdk/v65/objectstorage"` and
  `oci_common "github.com/oracle/oci-go-sdk/v65/common"`
  ([bucket resource L22-L24](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/objectstorage_bucket_resource.go#L22-L24)).

So the full chain is:
**internal OpenAPI/Swagger spec → `oci-go-sdk` (generated by `OracleSDKGenerator`,
`// Code generated. DO NOT EDIT.`) → `terraform-provider-oci` (no generation
header; internally scaffolded ?? verify) → Terraform schema.**
The provider does **not** re-derive anything from the raw spec; it consumes the
already-generated Go SDK request/response/enum types. **[EVIDENCE]**

## 3. Plugin SDK v2 vs Plugin Framework

**Finding: predominantly Plugin SDK v2; a minimal Plugin Framework footprint;
both served together via `terraform-plugin-mux` (protocol v5).** **[EVIDENCE]**

- `go.mod` pulls in **both** stacks plus the mux:
  `terraform-plugin-sdk/v2 v2.38.1`, `terraform-plugin-framework v1.16.1`,
  `terraform-plugin-framework-validators v0.19.0`, `terraform-plugin-go v0.29.0`,
  `terraform-plugin-mux v0.21.0` —
  [`go.mod`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/go.mod).
- **The mux is wired in `main.go`:** the SDKv2 provider
  (`provider.Provider().GRPCProvider`) and the Framework provider
  (`providerserver.NewProtocol5(provider.New())`) are combined with
  `tf5muxserver.NewMuxServer(...)` and served via `tf5server.Serve(...)`. See
  [`main.go` L68-L97](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/main.go#L68-L97).
- **Two provider definitions coexist** in
  [`internal/provider/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/provider):
  `provider.go` (SDKv2, `helper/schema`) and `provider_framework.go`
  (Plugin Framework, `provider/schema`), with separate registration files
  (`register_resource.go` / `register_datasource.go` vs
  `register_framework_datasource.go`).
  - SDKv2 provider imports `terraform-plugin-sdk/v2/helper/schema` —
    [`provider.go` L1-L40](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/provider/provider.go#L1-L40).
  - Framework provider implements `provider.Provider` over
    `terraform-plugin-framework/...` —
    [`provider_framework.go` L1-L45](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/provider/provider_framework.go#L1-L45).
- **Framework footprint is tiny.** The 143 packages under `internal/service/`
  are SDKv2 (`helper/schema`). The Plugin Framework is used in a separate
  directory
  [`internal/service-framework/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/service-framework),
  which contains a **single** service, `vault`, with a **single** data source:
  `vault_secret_version_data_source.go` (imports
  `terraform-plugin-framework/datasource`) —
  [header L1-L25](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service-framework/vault/vault_secret_version_data_source.go#L1-L25).
  **[INFERENCE]** This is an early/partial Framework migration; the bulk of the
  provider remains on SDKv2.

## 4. Generated/scaffolded vs hand-written split

| Layer | Location | Nature |
|---|---|---|
| Per-service resources & data sources (SDKv2) | [`internal/service/<svc>/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/service) (143 pkgs) | Uniform, SDK-mapped; **no gen header** — internally scaffolded ?? verify, hand-finished |
| Per-service SDK client wiring | [`internal/client/<svc>_clients.go`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/client) | Uniform factory pattern; same nature |
| Resource-discovery "hints"/overrides | `internal/service/<svc>/<svc>_export.go` | Scaffolded registration + **hand-written** override hooks |
| One Framework data source | [`internal/service-framework/vault/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/service-framework/vault) | Hand-written (Plugin Framework) |
| Provider setup / auth / config | [`internal/provider/provider.go`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/provider/provider.go), `provider_framework.go` | **Hand-written core** |
| Schema/CRUD helpers, retries, crud framework | [`internal/tfresource/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/tfresource), `internal/utils/`, `internal/globalvar/` | **Hand-written core** |
| Resource discovery / export engine | [`internal/commonexport/`](https://github.com/oracle/terraform-provider-oci/tree/5eab52881903a807d3875c92e314ab085b13c848/internal/commonexport), `internal/resourcediscovery/` | **Hand-written core** |
| Per-service `helpers_<svc>.go` | e.g. [`helpers_objectstorage.go`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/helpers_objectstorage.go) | **Hand-written** per-service extensions (mirrors SDK "transfer manager" pattern) |
| Entry point / mux | [`main.go`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/main.go) | **Hand-written core** |

This mirrors the SDK's split philosophy (generated service surface + a
hand-written `common`/`auth` core + per-service hand-written extensions), but
without the SDK's machine-enforced generated/hand-written boundary (no
`GEN_TARGETS`/`NON_GEN_TARGETS` Makefile, no `-generated`/`-extensions` modules).
In the provider, the boundary is **by directory/convention only**: `internal/
service/*` (service surface) vs `internal/{provider,tfresource,client,commonexport,
resourcediscovery,utils,globalvar}` (hand-written core).

### Concrete examples
1. **Object Storage bucket (`oci_objectstorage_bucket`).** SDKv2 `schema.Resource`
   with `Create/Read/Update/Delete` mapping to `oci-go-sdk` `CreateBucket` /
   `GetBucket` / `UpdateBucket` / `DeleteBucket`. Registered in
   [`register_resource.go`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/register_resource.go)
   via `tfresource.RegisterResource("oci_objectstorage_bucket", ObjectStorageBucketResource())`.
   Hand-written extension alongside: `helpers_objectstorage.go` (upload/retention
   plumbing).
2. **Object Storage export hints.** `objectstorage_export.go` wires discovery
   metadata and contains hand-authored override functions
   (`processObjectStorageNamespace`, `getObjectStorageNamespaceHCLDatasource`)
   labeled "Custom overrides…" —
   [L11-L40](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service/objectstorage/objectstorage_export.go#L11-L40).
3. **Vault secret-version data source.** The lone Plugin Framework component, in
   the separate `internal/service-framework/` tree —
   [`vault_secret_version_data_source.go`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/internal/service-framework/vault/vault_secret_version_data_source.go).

### Resource discovery ("export") — a notable extra capability
Beyond normal Terraform plugin mode, `main.go` exposes an `-command=export`
mode that introspects a compartment and **emits Terraform configuration + state**
for existing OCI resources (driven by the `*_export.go` hints and
`internal/resourcediscovery/`). See
[`main.go` L99-L210](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/main.go#L99-L210).
This is a *runtime* code generator (reverse: cloud → HCL), distinct from any
build-time provider generation. **[EVIDENCE]**

## 5. Release cadence — lockstep with the SDK?

**Finding: weekly, content-synchronized with the SDK, but on the provider's own
version line and typically one day behind the SDK cut.** **[EVIDENCE]**

- Provider CHANGELOG (own version line, weekly):
  - `8.16.0 (May 27, 2026)`, `8.15.0 (May 20, 2026)`, `8.14.0 (May 13, 2026)` —
    [`CHANGELOG.md`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/CHANGELOG.md).
- The OCI SDKs released `…05-26`, `…05-19`, `…05-12` (Go `65.116.0`, Python
  `2.176.0`, etc. — see `oracle/notes.md`). The provider's `8.16.0 (May 27)` is
  **one day after** the SDK's `2026-05-26` batch, and its `8.15.0 (May 20)`
  trails `…05-19`. So: same weekly rhythm, one cut behind.
- **Content tracks the SDK spec batch.** The provider's `Added` bullets are the
  same spec-driven feature list the SDKs ship — e.g. `8.16.0` lists "Support for
  ZeroETL Mirror Phase 4", GoldenGate, Data Science items that match the SDK
  changelog vocabulary. **[INFERENCE]** Same internal spec batch flows: spec →
  SDK release (week N) → provider release (week N + ~1 day), pinning the new SDK
  version in `go.mod` (`v65.116.0`). **[EVIDENCE for the version pin; INFERENCE
  for the causal pipeline.]**
- **NOT a shared version number** with the SDK: provider is `8.x`, SDK is
  `v65.x`. Lockstep is on *cadence/content*, not on *version*.

## Evidence vs inference — summary

- **Directly evidenced:** built on `oci-go-sdk/v65` (go.mod + imports);
  SDKv2-dominant with a 1-data-source Plugin Framework footprint, muxed via
  `terraform-plugin-mux` in `main.go`; **no `OracleSDKGenerator` / `Code
  generated. DO NOT EDIT.` markers anywhere** in the provider; uniform
  143-service layout with 1:1 SDK-mapped CRUD; per-service `*_export.go`
  override hooks; weekly CHANGELOG tracking the SDK on an independent version
  line; MPL-2.0; runtime `export` (cloud→HCL) generator.
- **Inferred:** that an internal generator scaffolds the per-service provider
  files (strongly suggested by structure + SDK lockstep, but **unmarked** in the
  output, so weaker than the SDK case); that the same internal spec batch drives
  both SDK and provider releases.
- **Open gaps (`?? verify`):**
  1. Whether a private provider generator exists and whether it reuses
     `OracleSDKGenerator` infrastructure — not determinable from public files
     (no generator, no header, no template in-repo).
  2. The internal CI/build orchestration (provider repo retains a legacy
     [`wercker.yml`](https://github.com/oracle/terraform-provider-oci/blob/5eab52881903a807d3875c92e314ab085b13c848/wercker.yml)
     and `ocibuild.conf`, but the real pipeline runs internally upstream of
     GitHub).
  3. Roadmap/scope of the SDKv2 → Plugin Framework migration (only `vault`
     secret-version moved so far).
