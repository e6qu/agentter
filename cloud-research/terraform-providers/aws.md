# Terraform AWS providers — `aws` & `awscc`

**Two providers for the same cloud, opposite build philosophies.** This is the
sharpest illustration of the hand-written↔generated spectrum.

Full research with all citations: [`../iac/terraform-aws.md`](../iac/terraform-aws.md).
Cross-layer context (how providers feed CDKTF/Pulumi): [`../IAC_CODEGEN.md`](../IAC_CODEGEN.md).

## `terraform-provider-aws` — hand-written + scaffolding codegen

**Build model: mostly hand-written Go, with generated scaffolding.** Resources and
their CRUD logic are authored by hand on top of the **AWS SDK for Go v2**; a
`internal/generate/` toolbox (~30 generators, run via `//go:generate`) emits only
the *boilerplate around* them.

- [`internal/generate/`](https://github.com/hashicorp/terraform-provider-aws/tree/e26a14e0d078679494673567ab79363a7c1a60a0/internal/generate)
  — generators for service-package registration, tag handling, paginators, tests.
- The service registry is built by
  [`servicepackage/main.go`](https://github.com/hashicorp/terraform-provider-aws/blob/e26a14e0d078679494673567ab79363a7c1a60a0/internal/generate/servicepackage/main.go),
  which **regex-scans `// @FrameworkResource` / `// @SDKResource` annotations**
  on the hand-written resource files to discover what to register. So the
  hand-written code is the source of truth; the generator just indexes it.
- Generated files carry `// Code generated by ... ; DO NOT EDIT.`; resource logic
  does not.

**Plugin framework:** mostly **Plugin SDK v2**, migrating resource-by-resource to
the **Plugin Framework** (the two are muxed). License: **MPL-2.0**.

## `terraform-provider-awscc` — fully generated

**Build model: fully code-generated** from CloudFormation resource schemas via the
AWS Cloud Control API. ~1,366 `*_resource_gen.go` files, no per-resource CRUD.

- Generators in
  [`internal/provider/generators/`](https://github.com/hashicorp/terraform-provider-awscc/tree/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/provider/generators)
  read CloudFormation resource type schemas (e.g.
  [`AWS_Logs_AccountPolicy.json`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/service/cloudformation/schemas/AWS_Logs_AccountPolicy.json)),
  parse them with HashiCorp's `aws-cloudformation-resource-schema-sdk-go`
  (`cfschema`), and render Plugin Framework Go via `codegen.Emitter`.
- Every resource (e.g.
  [`account_policy_resource_gen.go`](https://github.com/hashicorp/terraform-provider-awscc/blob/e9fd27e90f70843339c5a6a4296e8e78282e69c6/internal/aws/logs/account_policy_resource_gen.go))
  delegates CRUD to the generic `cloudcontrol` Create/Read/Update/Delete — no
  hand-written per-resource logic.

**Plugin framework:** **Plugin Framework** throughout. License: **MPL-2.0**.

## The key relationship
`awscc`'s input — the **CloudFormation resource type schemas** — is the *same*
source AWS CDK's L1 (`Cfn*`) constructs are generated from (see
[`../IAC_CODEGEN.md`](../IAC_CODEGEN.md)). So one spec source drives both a fully
generated Terraform provider and the CDK L1 layer. Meanwhile `terraform-provider-aws`,
for the same cloud, is hand-written — the two coexist because `awscc` gives
day-one coverage of any CloudFormation-modeled resource, while `aws` gives the
hand-tuned ergonomics most users rely on.
