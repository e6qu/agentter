# How the major Terraform providers are built

Build-model profiles for the five major cloud Terraform providers. Companion to
[`../IAC_CODEGEN.md`](../IAC_CODEGEN.md) (the IaC layer) and
[`../CODE_GENERATION.md`](../CODE_GENERATION.md) (the SDK layer). Full per-provider
research with all citations lives in `../iac/terraform-*.md`.

A Terraform provider is Go-only, but its **schema** is the input that CDKTF and the
Pulumi bridge generate *their* multi-language bindings from — so how each provider
is built matters one layer up too (see the "generation chain" in `../IAC_CODEGEN.md`).

## The spectrum — most generated → most hand-written

```
fully generated, regenerated each time   Google (Magic Modules), AWS awscc
generated SDK underneath, hand-written    Azure azurerm (pandora), OCI, AWS aws
   provider on top
schema-driven generic provider            Azure azapi
one-shot generated seed, then hand-owned  Alibaba alicloud
```

## Matrix

| Provider | Build model | Generator / source | SDK underneath | Plugin framework | License |
|---|---|---|---|---|---|
| [AWS `aws`](aws.md) | hand-written + **scaffolding** gen | `internal/generate` (boilerplate only) | AWS SDK for Go v2 | SDK v2 → Framework (muxed) | MPL-2.0 |
| [AWS `awscc`](aws.md) | **fully generated** | CloudFormation schemas → `codegen.Emitter` | Cloud Control API (generic CRUD) | Framework | MPL-2.0 |
| [Google `google`/`-beta`](google.md) | **fully generated** (regenerated) | Magic Modules MMv1 (YAML → Go templates) | calls Google REST | SDK v2 (+ some Framework) | Apache-2.0 / MPL-2.0 |
| [Azure `azurerm`](azure.md) | hand-written over a **generated SDK** | hand-written; SDK generated by `pandora` | `go-azure-sdk` (generated) | SDK v2 + Framework (muxed) | MPL-2.0 |
| [Azure `azapi`](azure.md) | **schema-driven** | embeds generated Azure types (Bicep types) | generic `azcore` | Framework | MPL-2.0 |
| [OCI `oci`](oci.md) | unmarked; likely **internal scaffolding** | no public generator marker | `oci-go-sdk/v65` (generated) | SDK v2 (tiny Framework) | MPL-2.0 |
| [Alibaba `alicloud`](alibaba.md) | **one-shot generated seed**, then hand-owned | external scaffold tool (marker deleted on edit) | mix (Darabonba + legacy + bespoke) | **Plugin SDK v1** (oldest) | MPL-2.0 |

## Cross-cutting findings

- **"Generated" means very different things.** Google *regenerates* the Go each
  time from YAML (the Go is a build artifact); AWS `awscc` fully generates from
  CloudFormation; AWS `aws` generates only *scaffolding* around hand-written
  resources; OCI looks scaffolded but ships unmarked, hand-reviewed source;
  Alibaba generates a *one-shot seed* and then **deletes the "generated" marker on
  first edit**, abandoning regeneration. So the same word spans
  "continuously-regenerated artifact" → "disposable starting template."

- **The generation often happens one layer down — in the SDK.** `azurerm` is
  hand-written but sits on the pandora-**generated** `go-azure-sdk`; `oci` sits on
  the **generated** `oci-go-sdk`; `aws` sits on the Smithy-**generated** AWS SDK
  for Go v2. The provider can be hand-written *because* a generated SDK does the
  protocol work.

- **Plugin-framework adoption is a maturity tell.** Alibaba is still on the
  original **Plugin SDK v1**; AWS/Azure/Google/OCI are on **SDK v2** migrating to
  the typed **Plugin Framework** (muxed); the fully-generated `awscc` and the
  schema-driven `azapi` are **Framework**-native.

- **License is uniform: MPL-2.0** for every provider (the Terraform-ecosystem
  norm) — the one variation is Google's *generator/config* repo (`magic-modules`),
  which is Apache-2.0 while its `third_party` Go copied into the provider is MPL-2.0.

- **Same spec, two consumers (AWS).** `awscc`'s CloudFormation-schema source is the
  *same* one AWS CDK's L1 constructs are generated from — one spec, a fully
  generated provider *and* a CDK layer (see `../IAC_CODEGEN.md`).
