# Next-generation AWS SDK for Python (v3) — Smithy → generated Python

> Pinned commits (verified via the GitHub API / `gh`):
> - `smithy-lang/smithy-python` @ `2472785bbb35ca6a462522791a1e21a0c8ba2acd` (branch `develop`)
> - `awslabs/aws-sdk-python` @ `bc78957aa97bf324dab6eefd202edfcf09e1d460` (branch `develop`)
>
> Both repos are **Apache-2.0**. Everything below is quoted from file contents
> at those SHAs; links are commit-pinned permalinks in the Links section.

## TL;DR

The v3 effort inverts boto3/botocore's architecture. Where botocore is
*data-driven* — it ships JSON service models and builds client classes
dynamically at runtime with `type()` (no per-service generated source) — the v3
SDK is *code-generated*: a Java-based Smithy generator emits static Python
source per service (`client.py`, `models.py`, `_private/schemas.py`, `config.py`,
`auth.py`), each carrying a `# Code generated by smithy-python-codegen DO NOT
EDIT.` header. This mirrors `aws-sdk-go-v2` (smithy-go) and `aws-sdk-js-v3`
(smithy-typescript).

Two repos make up the effort:

1. **`smithy-lang/smithy-python`** — the tooling: a Java/Gradle Smithy→Python
   *code generator* plus the **hand-written core runtime** packages
   (`smithy_core`, `smithy_http`, `smithy_aws_core`, `aws_sdk_signers`, …). It
   contains **no generated AWS clients**.
2. **`awslabs/aws-sdk-python`** — the published v3 SDK: per-service
   **distributed** pip packages (one package per client under `clients/`), the
   generated output of running the smithy-python generator over AWS Smithy
   models.

**Maturity: pre-1.0 / experimental on both sides** (see the Maturity section —
this is the load-bearing comparison point against boto3, which is production).

## 1. `smithy-lang/smithy-python` — generator + core runtime

### Repository shape

Top-level (`/blob/2472785.../`): `codegen/` (the Java generator, Gradle) and
`packages/` (the hand-written Python runtime). The README is explicit that the
repo holds **two components and no generated clients**:

> This repository contains two major components:
> 1) Smithy code generators for Python
> 2) Core modules and interfaces for building service clients in Python
> …
> This repository does *not* contain any generated clients, such as for S3 or
> other AWS services. Rather, these are the tools that facilitate the
> generation of those clients and non-AWS Smithy clients.
> — `README.md`

### Status quote (experimental / pre-1.0)

The README leads with an unambiguous warning:

> **WARNING: All interfaces are subject to change.**
>
> We are in the early stages of beginning work on low-level Python SDK
> modules… This code generator, and the clients it generates, are unstable and
> should not be used in production systems yet. Several features, such as
> detailed logging, have not been implemented yet.
> — `README.md`

The codegen artifacts are versioned `0.3.0` (`group = "software.amazon.smithy.python"`,
`version = "0.3.0"` in `codegen/build.gradle.kts`), and the README's example
`smithy-build.json` pins generator deps at `0.0.1` — i.e. pre-1.0.

### The code-generation module (Java, Gradle, writer-based)

`codegen/` is a Gradle multi-module Java project. `codegen/settings.gradle.kts`
declares the modules:

```
rootProject.name = "smithy-python"
include(":core")
include(":protocol-test")
include(":aws:core")
include(":plugins")
include(":plugins:types")
```

- **`:core`** (`codegen/core/`) — the core generator. `build.gradle.kts` sets
  `group = "software.amazon.smithy.python.codegen"` and depends on
  `api(libs.smithy.codegen)` (Smithy's `smithy-codegen-core`), plus
  `smithy-waiters`, `smithy-protocol-test-traits`, and `smithy-aws-traits`
  ("because we're using RestJson1 as a 'generic' protocol").
- **`:aws:core`** (`codegen/aws/core/`) — AWS-specific integrations
  (`AwsAuthIntegration`, `AwsProtocolsIntegration`, `AwsQueryProtocolGenerator`,
  `AwsStandardRegionalEndpointsIntegration`, `AwsUserAgentIntegration`, …).
- **`:plugins:types`** (`codegen/plugins/types/`) — the `python-type-codegen`
  plugin that emits only shape classes (no client).

**Entry point — Smithy build plugin.** `PythonClientCodegenPlugin` implements
Smithy's `SmithyBuildPlugin`; its plugin name is the one referenced in
`smithy-build.json`:

```java
public final class PythonClientCodegenPlugin implements SmithyBuildPlugin {
    @Override
    public String getName() {
        return "python-client-codegen";
    }
```
— `codegen/core/.../PythonClientCodegenPlugin.java`

**Directed codegen.** The orchestration uses Smithy's `DirectedCodegen`
interface (shared driver in `smithy-codegen-core`), not bespoke templating:

```java
final class DirectedPythonClientCodegen
        implements DirectedCodegen<GenerationContext, PythonSettings, PythonIntegration> {
    …
    public void generateService(GenerateServiceDirective<…> directive) { … }
    public void generateStructure(GenerateStructureDirective<…> directive) { … }
```
— `codegen/core/.../DirectedPythonClientCodegen.java`

**Writer-based (not Handlebars/Mustache templates).** The generator uses
Smithy's `SymbolWriter` abstraction. `PythonWriter extends
SymbolWriter<PythonWriter, ImportDeclarations>` and registers custom formatters
(`$T` for symbols, `$N` for nodes) plus auto-managed imports:

```java
@SmithyUnstableApi
public final class PythonWriter extends SymbolWriter<PythonWriter, ImportDeclarations> {
    …
    putFormatter('T', new PythonSymbolFormatter());
    putFormatter('N', new PythonNodeFormatter(this));
```
— `codegen/core/.../writer/PythonWriter.java`

Generators emit code by calling `writer.write("""…""")` with Java text blocks
and `writer.addImport(...)`/`pushState()`/`popState()`, e.g. in
`StructureGenerator.run()`:

```java
public void run() { … writer.write(""" … """); … }
…
writer.addImport("smithy_core.serializers", "ShapeSerializer");
writer.write("def serialize_members(self, serializer: ShapeSerializer):").indent();
```
— `codegen/core/.../generators/StructureGenerator.java` (lines ~82–383)

Per-shape generators live under `codegen/core/.../generators/`
(`StructureGenerator`, `UnionGenerator`, `EnumGenerator`, `OperationGenerator`,
`SchemaGenerator`, `ConfigGenerator`, `SetupGenerator` for `pyproject.toml`,
etc.). Protocols and runtime customizations are pluggable via
`PythonIntegration` (SPI on the classpath); `RestJsonIntegration` is the
reference protocol implementation.

The generator shells out to **`ruff`** (format) and **`pyright`** (type-check)
if present, and uses **`pandoc`** to convert Smithy doc traits into
Google-style docstrings (per `README.md` / `codegen/README.md`). Requires JDK
17+.

### Hand-written core runtime (`packages/`)

These are the transport/auth/serialization building blocks that generated
clients import — the hand-authored counterpart to the generated source. Each is
its own pip package:

| Package | Purpose |
|---|---|
| `smithy-core` | transport-agnostic core: schemas, serializers/deserializers, retries, interceptors, endpoints, auth/identity interfaces, event streams |
| `smithy-http` | HTTP core + async clients (`aiohttp`, `awscrt`), restJson serde, user-agent interceptor |
| `smithy-aws-core` | AWS implementations of core interfaces (SigV4, AWS protocols, event-stream) |
| `aws-sdk-signers` | standalone SigV4 request signers (no client needed) |
| `smithy-json`, `smithy-xml`, `smithy-aws-event-stream` | wire-format codecs |

Representative hand-written modules (`packages/smithy-core/src/smithy_core/`):
`schemas.py`, `serializers.py`, `deserializers.py`, `retries.py`,
`interceptors.py`, `auth.py`, `aio/client.py` (the `ClientCall` /
`RequestPipeline` the generated `client.py` calls), `aio/eventstream.py`,
`aio/interfaces/{auth,identity,eventstream}.py`.

README confirms the split and the design intent ("Components must be typed…
usable via `pyright`"; consistency with smithy-typescript / smithy-go):

> * `smithy-core` provides transport-agnostic core modules and interfaces…
> * `smithy-http` provides HTTP core modules… two async HTTP clients usable
>   with the `aiohttp` or `awscrt` optional dependency sets…
> * `smithy-aws-core` provides implementations of `smithy-core` interfaces for
>   AWS, such as SigV4 signers.
> — `README.md`

`aws-sdk-signers` is independently shipped and labelled **Alpha**:

> This project is currently in an **Alpha** phase of development. There likely
> will be breakages and redesigns between minor patch versions…
> — `packages/aws-sdk-signers/README.md`

(its `pyproject.toml` classifier: `"Development Status :: 2 - Pre-Alpha"`.)

## 2. `awslabs/aws-sdk-python` — the published v3 SDK

### Repository shape & distribution model

`/blob/bc78957.../`:
- `clients/` — one directory **per service package**.
- `codegen/aws-models/` — the input AWS Smithy model JSONs
  (`bedrock-runtime.json`, `polly.json`, `transcribe-streaming.json`, etc.) fed
  to the smithy-python generator.

At this SHA `clients/` holds 7 packages (the in-preview surface):
`aws-sdk-bedrock-runtime`, `aws-sdk-connecthealth`, `aws-sdk-lex-runtime-v2`,
`aws-sdk-polly`, `aws-sdk-sagemaker-runtime-http2`,
`aws-sdk-transcribe-streaming`, plus `aws-sdk-python` (a meta/umbrella package).

Each service package is a self-contained pip distribution — e.g.
`clients/aws-sdk-polly/` has its own `pyproject.toml`, `LICENSE`, `NOTICE`,
`CHANGELOG.md`, `README.md`, `docs/`, `tests/`, and
`src/aws_sdk_polly/{__init__,client,models,config,auth,user_agent}.py` plus
`_private/schemas.py`. This is the explicit anti-botocore design:

> Unlike Boto3, these clients are distributed per-service, leaving you the
> option to pick what fits your needs.
> — `README.md`

The umbrella **`aws-sdk-python`** package has `dependencies = []` and exposes
each client only as an **optional extra**, so installing it pulls in nothing by
default:

```toml
name = "aws-sdk-python"
description = "Meta-package containing all AWS service clients."
dependencies = []

[project.optional-dependencies]
bedrock_runtime = ["aws_sdk_bedrock_runtime==0.6.0"]
sagemaker_runtime_http2 = ["aws_sdk_sagemaker_runtime_http2==0.6.0"]
transcribe_streaming = ["aws_sdk_transcribe_streaming==0.6.0"]
all = ["aws_sdk_python[bedrock_runtime]", …]
```
— `clients/aws-sdk-python/pyproject.toml`

Contrast botocore: a **single mega-package** that bundles every service's JSON
model and synthesizes clients at runtime.

### A generated client file + DO-NOT-EDIT header

`clients/aws-sdk-polly/src/aws_sdk_polly/client.py` (line 1):

```python
# Code generated by smithy-python-codegen DO NOT EDIT.

from copy import deepcopy
import logging

from smithy_core.aio.client import ClientCall, RequestPipeline
from smithy_core.aio.eventstream import DuplexEventStream
from smithy_core.interceptors import InterceptorChain
from smithy_core.retries import RetryStrategyResolver
from smithy_http.plugins import user_agent_plugin

from .config import Config, Plugin
from .models import (DELETE_LEXICON, DescribeVoicesInput, …)
```

The same header appears on `models.py`, `_private/schemas.py`, and even the
generated `pyproject.toml`:

```python
# Code generated by smithy-python-codegen DO NOT EDIT.
[project]
name = "aws_sdk_polly"
version = "0.6.0"
…
classifiers = [ "Development Status :: 2 - Pre-Alpha", … ]
dependencies = [
    "smithy_aws_core[eventstream, json]~=0.6.0",
    "smithy_core~=0.5.0",
    "smithy_http[awscrt]~=0.4.0",
]
```
— `clients/aws-sdk-polly/pyproject.toml`

Generated code is **async** (`async def`, `await client.echo_message(...)` in
the README example) and **typed** (dataclass input/output models, `StrEnum`s,
`Self`, `Literal`).

### Schema-driven generated source (the boto3 inversion, concretely)

The interesting nuance: botocore interprets a JSON model *at runtime*. v3
instead generates Python source that **constructs the equivalent schema objects
statically**, which the hand-written `smithy_core` serializers then drive. See
the generated `_private/schemas.py`:

```python
# Code generated by smithy-python-codegen DO NOT EDIT.
from smithy_core.schemas import Schema
from smithy_core.shapes import ShapeID, ShapeType
from smithy_core.traits import Trait

ALPHABET = Schema(id=ShapeID("com.amazonaws.polly#Alphabet"), shape_type=ShapeType.STRING)

AUDIO_EVENT = Schema.collection(
    id=ShapeID("com.amazonaws.polly#AudioEvent"),
    members={"AudioChunk": {"target": AUDIO_CHUNK,
        "traits": [Trait.new(id=ShapeID("smithy.api#eventPayload"))]}},
)
```
— `clients/aws-sdk-polly/src/aws_sdk_polly/_private/schemas.py`

So the runtime is still partly schema-driven, but the schema is *materialized as
generated, importable, typed Python* rather than parsed from a shipped JSON blob
and reflected into dynamic classes.

## 3. Build model & generated-vs-hand-written split

| Aspect | smithy-python (tooling) | aws-sdk-python (output) |
|---|---|---|
| Language / build | Java + Gradle multi-module (`:core`, `:aws:core`, `:plugins:types`) | per-package `pyproject.toml` + **hatchling** build backend |
| Codegen style | **writer-based** (`SymbolWriter`/`PythonWriter`, Java text blocks, `$T`/`$N` formatters, auto-import management) — not Handlebars/templates | n/a (consumes generator) |
| Orchestration | Smithy `DirectedCodegen` driver; `python-client-codegen` `SmithyBuildPlugin`; pluggable `PythonIntegration` SPI for protocols/customizations | `smithy build` over `codegen/aws-models/*.json` |
| Post-processing | `ruff` format + `pyright` check (if on PATH); `pandoc` for docstrings | committed, formatted generated source |

**Generated** (per service, DO-NOT-EDIT): `client.py` (async operations,
pipeline wiring), `models.py` (dataclass I/O shapes, enums, error types),
`_private/schemas.py` (static `Schema` objects), `config.py`, `auth.py`,
`user_agent.py`, and the package `pyproject.toml`/README/docs stubs.

**Hand-written** (shared, in smithy-python `packages/`): the entire runtime —
HTTP transport (`smithy_http`, aiohttp/awscrt clients), retries
(`smithy_core.retries`), interceptors/plugins, auth & SigV4 signing
(`smithy_aws_core`, `aws_sdk_signers`), serialization engines
(`serializers`/`deserializers`, `smithy_json`/`smithy_xml`), event streams,
endpoint resolution. Generated clients depend on these via normal pip deps.

This is the same generated-client + hand-written-core division used by
`aws-sdk-go-v2`/`smithy-go` and `aws-sdk-js-v3`/`smithy-typescript`; the
smithy-python README explicitly says it "will heavily draw inspiration from the
precedents set by smithy-typescript and smithy-go."

## 4. Comparison data points vs boto3/botocore

| Dimension | boto3 / botocore (current, production) | aws-sdk-python v3 (pre-1.0) |
|---|---|---|
| Client class creation | **Runtime synthesis** — classes built dynamically with `type()`; no per-service generated source files | **Static generated source** — `client.py` committed per service, `# Code generated … DO NOT EDIT.` |
| Service model | One JSON model per service, interpreted at runtime, all bundled in **one botocore package** | Smithy model → generated static `Schema` objects (`_private/schemas.py`); model lives in the generator's input, not shipped as runtime JSON |
| Packaging | **Single mega-package** (botocore bundles all services) | **One pip package per service** under `clients/`; umbrella `aws-sdk-python` pulls clients only via extras |
| Typing | Largely dynamic; types via separate stub packages (boto3-stubs/mypy-boto3) | **Typed by construction** — dataclasses, `StrEnum`, `Self`, `Literal`; generator runs `pyright --strict` |
| Sync/async | Sync (boto3); async via 3rd-party aiobotocore | **Async-first** (`async def`, asyncio) |
| Runtime core | botocore (hand-written, monolithic) | hand-written but **modular** (`smithy_core`/`smithy_http`/`smithy_aws_core`/`aws_sdk_signers`), reusable beyond AWS |
| Maturity | **Production / GA**, stable public API | **Pre-1.0 / experimental**; "should not be used in production systems yet"; breaking changes between minor versions; Pre-Alpha classifiers; ~7 preview services |

**Precise maturity statement:** As of the pinned commits (May 2026), v3 is an
explicitly experimental preview — generator artifacts at `0.3.0`, client
packages at `0.6.0`, runtime packages at `0.x`, all classified
`Development Status :: 2 - Pre-Alpha`, with only a handful of (mostly
streaming-oriented: bedrock-runtime, polly, transcribe-streaming,
sagemaker-runtime, lex-runtime, connecthealth) services published. boto3 remains
the production SDK; v3 does not replace it yet.
