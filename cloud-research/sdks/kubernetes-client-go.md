# Kubernetes Client Libraries: `client-go` and the Dual Codegen Model

## Summary

Kubernetes is unusual among the providers surveyed: it does **not** use a single
cross-language SDK generator. Instead it runs **two completely different code
generation models** in parallel, and the difference is the headline finding.

1. **The Go client (`k8s.io/client-go`) is generated from hand-written Go API
   types** — *not* from OpenAPI. The authoritative source of the API is a tree of
   hand-written Go structs in `k8s.io/api`, annotated with `// +...` *marker
   comments*. A suite of generators in `k8s.io/code-generator`
   (`client-gen`, `lister-gen`, `informer-gen`, `deepcopy-gen`, `conversion-gen`,
   `defaulter-gen`, `applyconfiguration-gen`, ...) — all built on the
   `k8s.io/gengo` Go-AST framework — read those types and their markers and emit
   the typed clientsets, listers, informers, deepcopy methods, etc.

2. **The other official clients (`python`, `java`, `javascript`, `csharp`,
   `ruby`, etc.) are generated from the published Kubernetes OpenAPI/Swagger
   spec** via **OpenAPI Generator** (OpenAPITools), orchestrated by the
   `kubernetes-client/gen` repo. The spec itself
   (`api/openapi-spec/swagger.json` in `kubernetes/kubernetes`) is produced by
   the apiserver's OpenAPI machinery from the *same* Go types — so OpenAPI is a
   downstream artifact of the Go types, not the source for the Go client.

So the same hand-written Go structs are the root of *both* pipelines, but the Go
client branches off **before** OpenAPI (directly off the Go AST via gengo), while
every other language branches off **after** OpenAPI (off the swagger.json via
openapi-generator). All repositories are **Apache-2.0** licensed.

## 1. The Go client: gengo-from-Go-types

### 1.1 The source is hand-written Go structs with marker comments

The API surface is defined as ordinary, hand-written Go structs in
`k8s.io/api`. Each "root" type that should get a typed client is annotated with a
`// +genclient` marker comment, plus `// +k8s:deepcopy-gen` and
`// +k8s:prerelease-lifecycle-gen` markers. These comments are the contract that
drives generation — they are read by the generators, not by the Go compiler.

For example, `apps/v1.Deployment` carries a rich set of markers, including
subresource directives that tell `client-gen` to emit `GetScale`/`UpdateScale`/
`ApplyScale` methods against the `scale` subresource
([`apps/v1/types.go` L351–359](https://github.com/kubernetes/api/blob/e622342ab4abd51157f8fd650186575bab46777c/apps/v1/types.go#L351-L359)):

```go
// +genclient
// +genclient:method=GetScale,verb=get,subresource=scale,result=k8s.io/api/autoscaling/v1.Scale
// +genclient:method=UpdateScale,verb=update,subresource=scale,input=k8s.io/api/autoscaling/v1.Scale,result=k8s.io/api/autoscaling/v1.Scale
// +genclient:method=ApplyScale,verb=apply,subresource=scale,input=k8s.io/api/autoscaling/v1.Scale,result=k8s.io/api/autoscaling/v1.Scale
// +k8s:deepcopy-gen:interfaces=k8s.io/apimachinery/pkg/runtime.Object
// +k8s:prerelease-lifecycle-gen:introduced=1.9

// Deployment enables declarative updates for Pods and ReplicaSets.
type Deployment struct {
```

Cluster-scoped types add `// +genclient:nonNamespaced`, e.g.
`rbac/v1.ClusterRole`
([`rbac/v1/types.go` L176–178](https://github.com/kubernetes/api/blob/e622342ab4abd51157f8fd650186575bab46777c/rbac/v1/types.go#L176-L178)):

```go
// +genclient
// +genclient:nonNamespaced
// +k8s:deepcopy-gen:interfaces=k8s.io/apimachinery/pkg/runtime.Object
```

This tells `client-gen` to omit the namespace from the generated client's method
signatures and request paths.

### 1.2 The generators live in `k8s.io/code-generator` and are built on `k8s.io/gengo`

The `kubernetes/code-generator` repo ships one `cmd/<name>-gen` binary per
generator
([`cmd/` listing](https://github.com/kubernetes/code-generator/tree/600daa23ce07e12253b953b279df498c5e7014e1/cmd)):

| Generator | Output | DO-NOT-EDIT marker emitted |
|---|---|---|
| `client-gen` | typed clientsets (`client-go/kubernetes/typed/...`) | `// Code generated by client-gen. DO NOT EDIT.` |
| `lister-gen` | listers (`client-go/listers/...`) | `// Code generated by lister-gen. DO NOT EDIT.` |
| `informer-gen` | informers (`client-go/informers/...`) | `// Code generated by informer-gen. DO NOT EDIT.` |
| `deepcopy-gen` | `zz_generated.deepcopy.go` (in `k8s.io/api`) | `// Code generated by deepcopy-gen. DO NOT EDIT.` |
| `conversion-gen` | internal↔versioned conversions | `// Code generated by conversion-gen. DO NOT EDIT.` |
| `defaulter-gen` | defaulting funcs | `// Code generated by defaulter-gen. DO NOT EDIT.` |
| `applyconfiguration-gen` | server-side-apply builders (`client-go/applyconfigurations/...`) | (generated) |
| `register-gen` / `prerelease-lifecycle-gen` / `go-to-protobuf` | scheme registration, lifecycle, protobuf marshalers | (generated) |

All of them are thin front-ends over `k8s.io/gengo/v2`, the shared Go-AST
parsing + templating framework. `code-generator`'s `go.mod` declares the
dependency
([`go.mod`](https://github.com/kubernetes/code-generator/blob/600daa23ce07e12253b953b279df498c5e7014e1/go.mod)):

```
k8s.io/gengo/v2 v2.0.0-20240911193312-2b36238f13e9
```

and each generator wires gengo up directly. `client-gen`'s `main.go` is explicit
about this ([`cmd/client-gen/main.go` L17, L29–58](https://github.com/kubernetes/code-generator/blob/600daa23ce07e12253b953b279df498c5e7014e1/cmd/client-gen/main.go#L17-L58)):

```go
// client-gen makes the individual typed clients using gengo.
...
	"k8s.io/gengo/v2"
	"k8s.io/gengo/v2/generator"
...
	if err := gengo.Execute(
```

gengo parses the Go packages, exposes the parsed types plus their extracted
marker comments, and runs each generator's templates over them.

The typed clientset for Deployments, generated by `client-gen` from the
`Deployment` type above, opens with the canonical generated-code banner
([`kubernetes/typed/apps/v1/deployment.go` L17–25](https://github.com/kubernetes/client-go/blob/0d554618e21a4605cafd4264a685b916d2c024e9/kubernetes/typed/apps/v1/deployment.go#L17-L25)):

```go

package v1

import (
	context "context"
	fmt "fmt"

	appsv1 "k8s.io/api/apps/v1"
```

The matching lister and informer carry their own banners —
[`listers/apps/v1/deployment.go` L17](https://github.com/kubernetes/client-go/blob/0d554618e21a4605cafd4264a685b916d2c024e9/listers/apps/v1/deployment.go#L17)
(`// Code generated by lister-gen. DO NOT EDIT.`) and
[`informers/apps/v1/deployment.go` L17](https://github.com/kubernetes/client-go/blob/0d554618e21a4605cafd4264a685b916d2c024e9/informers/apps/v1/deployment.go#L17)
(`// Code generated by informer-gen. DO NOT EDIT.`).
The deepcopy methods land back in `k8s.io/api` itself, e.g.
[`apps/v1/zz_generated.deepcopy.go` L20](https://github.com/kubernetes/api/blob/e622342ab4abd51157f8fd650186575bab46777c/apps/v1/zz_generated.deepcopy.go#L20)
(`// Code generated by deepcopy-gen. DO NOT EDIT.`).

### 1.4 Orchestration: `kube_codegen.sh`

The pipeline is driven by `kube_codegen.sh`, a shell library shipped in
`code-generator` that builds the `*-gen` binaries and runs them over a module's
packages. Its `gen_helpers` function runs `deepcopy-gen` (selecting packages that
contain a `// +k8s:deepcopy-gen=` comment), and `gen_client` runs `client-gen`,
`lister-gen`, and `informer-gen`, regenerating files that match the
`// Code generated by ... DO NOT EDIT.` markers
([`kube_codegen.sh` L107, L555–666](https://github.com/kubernetes/code-generator/blob/600daa23ce07e12253b953b279df498c5e7014e1/kube_codegen.sh#L555-L666)).
In the main `kubernetes/kubernetes` build this is invoked from
`hack/update-codegen.sh`; downstream CRD/operator projects (controller-runtime,
Operator SDK, etc.) call the same library to generate clients for their own
`+genclient`-marked types. (Older entrypoints `generate-groups.sh` /
`generate-internal-groups.sh` are still present for backward compatibility.)

## 2. The other clients: OpenAPI Generator off the swagger spec

### 2.1 `kubernetes-client/gen` is the orchestrator

Every non-Go official client is generated by a shared driver repo,
`kubernetes-client/gen`. Its `openapi/` directory holds one `<lang>.sh` +
`<lang>.xml` pair per language and a common `openapi-generator/` driver
([`openapi/` listing](https://github.com/kubernetes-client/gen/tree/11a6ddfdfae2506c7917f4b20576278364e88df1/openapi)):
`python.sh`, `java.sh`, `csharp.sh`, `typescript.sh`/`typescript-fetch.sh`,
`ruby.sh`, `go.sh`, `rust.sh`, `haskell.sh`, `perl.sh`, plus the spec
preprocessor `preprocess_spec.py`.

The Python driver `openapi/python.sh` sources the common generator, pins an
OpenAPI Generator version, and post-processes the output (e.g. rewriting imports
to the `kubernetes.client` package)
([`openapi/python.sh`](https://github.com/kubernetes-client/gen/blob/11a6ddfdfae2506c7917f4b20576278364e88df1/openapi/python.sh)):

```sh
source "${SCRIPT_ROOT}/openapi-generator/client-generator.sh"
source "${SETTING_FILE}"
OPENAPI_GENERATOR_COMMIT="${OPENAPI_GENERATOR_COMMIT:-v6.6.0}"

CLIENT_LANGUAGE=python; \
CLEANUP_DIRS=(client/api client/apis client/models docs test); \
kubeclient::generator::generate_client "${OUTPUT_DIR}"
```

### 2.2 The actual generation runs OpenAPI Generator (OpenAPITools) via Maven

`generate_client` builds a Docker image and runs
`generate_client_in_container.sh` inside it. That script (a) preprocesses the
spec, then (b) invokes the **`openapi-generator-maven-plugin`** (group
`org.openapitools`) through `mvn ... generate-sources`
([`openapi/openapi-generator/generate_client_in_container.sh` L57, L82, L92–105](https://github.com/kubernetes-client/gen/blob/11a6ddfdfae2506c7917f4b20576278364e88df1/openapi/openapi-generator/generate_client_in_container.sh#L57-L105)):

```sh
FOLDERS=(/root/.m2/repository/org/openapitools/openapi-generator-maven-plugin/*)
...
python3 "${SCRIPT_ROOT}/preprocess_spec.py" "${CLIENT_LANGUAGE}" "${KUBERNETES_BRANCH}" "${output_dir}/swagger.json" ...
...
mvn_args=(
    -Dgenerator.spec.path="${output_dir}/swagger.json"
    ...
)
mvn -f "${SCRIPT_ROOT}/generation_params.xml" clean generate-sources "${mvn_args[@]}"
```

### 2.3 The spec source is `kubernetes/kubernetes`'s published swagger.json

`preprocess_spec.py` downloads the spec straight from the main Kubernetes repo by
branch/tag, then mutates it (collapsing `watch` operations into
`?watch=true`, injecting custom-objects paths, etc.) before generation
([`openapi/preprocess_spec.py` L564–605](https://github.com/kubernetes-client/gen/blob/11a6ddfdfae2506c7917f4b20576278364e88df1/openapi/preprocess_spec.py#L564-L605)):

```python
spec_url = 'https://raw.githubusercontent.com/%s/%s/' \
       '%s/api/openapi-spec/swagger.json' % (args.username,
                                             ...
                                             args.kubernetes_branch)
```

That `api/openapi-spec/swagger.json` is itself a generated artifact: the
apiserver emits it from the same Go types described in section 1. So OpenAPI sits
*downstream* of the Go types for these clients.

### 2.4 The generated output proves the generator

A generated Python model is unmistakably OpenAPI Generator output
([`kubernetes/client/models/v1_deployment.py` L1–8](https://github.com/kubernetes-client/python/blob/ba5d42e4ce8c3d8c4d383028e4d140d942a6b5ca/kubernetes/client/models/v1_deployment.py#L1-L8)):

```python
# coding: utf-8

"""
    Kubernetes

    No description provided (generated by Openapi Generator https://github.com/openapitools/openapi-generator)  # noqa: E501

    The version of the OpenAPI document: release-1.36
    Generated by: https://openapi-generator.tech
"""
```

The Java client is the same story — its generated models live in the
`io.kubernetes.client.openapi.models` package (note the `.openapi.` segment),
e.g.
[`V1Deployment.java`](https://github.com/kubernetes-client/java/blob/99cc0b6b8da119b10c38cd7b15fc736405f5d1ff/kubernetes/src/main/java/io/kubernetes/client/openapi/models/V1Deployment.java).
`javascript` (typescript-fetch templates) and `csharp` follow the identical
`gen`-repo + openapi-generator path via `typescript.sh`/`csharp.sh`.

Each language repo keeps the `gen`-produced code under a `kubernetes/` (Python),
`kubernetes/src/main/java/io/kubernetes/client/openapi/` (Java), `src/gen/`
(JS/TS), etc. subtree, with thin hand-written ergonomic layers on top
(authentication/kubeconfig loading, `watch`, `stream`/exec helpers) that are
*not* generated. The Python repo applies a few post-gen patches/diffs in
`scripts/` (`api_client_patch.diff`, `rest_client_patch.diff`, ...) on top of the
openapi-generator output
([`kubernetes-client/python/scripts`](https://github.com/kubernetes-client/python/tree/ba5d42e4ce8c3d8c4d383028e4d140d942a6b5ca/scripts)).

## 3. Generated vs hand-written

### client-go

| Generated (gengo) | Hand-written (the runtime "machinery") |
|---|---|
| `kubernetes/` typed clientsets (`client-gen`) | `rest/` — REST client, request builder, transport config |
| `listers/` (`lister-gen`) | `transport/` — round-trippers, TLS, auth, bearer-token/cert |
| `informers/` (`informer-gen`) | `tools/cache/` — shared informers, reflector, DeltaFIFO, indexers |
| `applyconfigurations/` (`applyconfiguration-gen`) | `tools/clientcmd/`, `tools/leaderelection/`, `tools/record/`, `tools/portforward/` |
| `scheme/` registration (`register-gen`) | `discovery/`, `dynamic/`, `metadata/`, `scale/`, `plugin/` (auth plugins) |
| deepcopy/conversion/defaults in `k8s.io/api` | `util/` (workqueue, retry, flowcontrol) |

The hand-written files carry a plain `package` declaration with **no**
`DO NOT EDIT` banner — e.g.
[`rest/client.go`](https://github.com/kubernetes/client-go/blob/0d554618e21a4605cafd4264a685b916d2c024e9/rest/client.go)
and
[`tools/cache/shared_informer.go`](https://github.com/kubernetes/client-go/blob/0d554618e21a4605cafd4264a685b916d2c024e9/tools/cache/shared_informer.go)
([top-level dir listing](https://github.com/kubernetes/client-go/tree/0d554618e21a4605cafd4264a685b916d2c024e9)).
The generated typed layer *uses* this hand-written machinery: a generated client
method ultimately builds a `rest.Request` and a generated informer is wired on
top of `tools/cache`'s shared informer + reflector. **The source of truth for the
whole thing is the hand-written Go API types in `k8s.io/api`.**

### OpenAPI clients

The model classes, API/operation classes, and serialization scaffolding are
generated by openapi-generator from swagger.json. Hand-written on top: config /
kubeconfig loading, auth (exec credential plugins, token refresh), `watch`
streaming, exec/attach/port-forward stream helpers, and the post-gen patch diffs.

## 4. Multi-language strategy

There is **no single cross-language generator** for Kubernetes clients, and that
is deliberate:

- **Go** is generated directly off the Go AST via `gengo`/`code-generator`,
  because the API *is* defined in Go and the Go client needs idiomatic,
  strongly-typed clientsets/informers/listers that a generic OpenAPI generator
  could not produce as cleanly (server-side-apply builders, subresource methods,
  the informer/lister cache architecture).
- **Everything else** (python/java/javascript/csharp/ruby/...) is generated off
  the published `swagger.json` via **OpenAPI Generator**, orchestrated centrally
  by `kubernetes-client/gen`, so adding a language is "add a `<lang>.sh` +
  `<lang>.xml` and lean on openapi-generator's existing language support."

Both pipelines ultimately trace back to the same hand-written Go structs in
`k8s.io/api`; they simply branch at different points — Go before OpenAPI (off the
AST), all others after OpenAPI (off swagger.json).

## Gaps / `?? verify`

- **`api/openapi-spec/swagger.json` provenance** — I confirmed the `gen` repo
  *downloads* it from `kubernetes/kubernetes`, but did not open the apiserver
  OpenAPI-emission code (`k8s.io/kube-openapi`, `hack/update-openapi-spec.sh`)
  that *produces* it from the Go types. The "OpenAPI is downstream of the Go
  types" claim is well-established but not pinned to that emitter here. `?? verify`
- **OpenAPI Generator version drift** — `python.sh` defaults to
  `OPENAPI_GENERATOR_COMMIT=v6.6.0`, but the actual plugin version is auto-detected
  from the Maven repo inside the build container; the pinned value may differ per
  language/`SETTING_FILE`. Treat v6.6.0 as the Python reference, not a global pin.
  `?? verify`
- **client-go v12 vs v0.x tags** — client-go publishes both legacy `vN.0.0`
  (e.g. `v12.0.0`) and `v0.<minor>.0` tags. This doc pins the `v0.32.0`
  (Kubernetes 1.32) line for cross-repo consistency. Newer releases (1.36) exist;
  generation model is unchanged.
