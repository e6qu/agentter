# AWS SDK for Go v2 (`aws-sdk-go-v2`)

**Build model: codegen from a type-system IDL (Smithy).** Each `service/<svc>` Go
module is generated from a Smithy model by a writer-based generator; the runtime
is the hand-written `smithy-go` middleware stack plus `aws/` core.

Per-SDK profile; full AWS analysis in
[`../CODE_GENERATION.md`](../CODE_GENERATION.md). The AWS local reproduction used
the TypeScript target ([`../aws/runs/sqs-typescript/`](../aws/runs/sqs-typescript/README.md)),
but the Go pipeline is the same shape with smithy-go.

## Source model
**Smithy 2.0 JSON AST**, vendored at
[`codegen/sdk-codegen/aws-models/`](https://github.com/aws/aws-sdk-go-v2/tree/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/codegen/sdk-codegen/aws-models)
and synced from the public model source via the Makefile's `sync-api-models`.

## Generator + engine
[`aws/smithy-go`](https://github.com/aws/smithy-go/tree/6857390fd160b416d0e369dc9dc5c2b3736178fc/codegen/smithy-go-codegen/src/main/java/software/amazon/smithy/go/codegen)
— a **Java** codegen project using `GoWriter` / `AbstractDirectedCodegen`
(writer-based, not templated). The AWS layer is `smithy-aws-go-codegen` in the
repo's `codegen/` Gradle project. Generation is driven from the
[Makefile](https://github.com/aws/aws-sdk-go-v2/blob/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/Makefile)
(`make generate`, single-service `SMITHY_GO_BUILD_API=... make generate-dev`),
with the codegen commit pinned in
[`SMITHY_GO_CODEGEN_VERSION`](https://github.com/aws/aws-sdk-go-v2/blob/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/SMITHY_GO_CODEGEN_VERSION).
Generated files carry `// Code generated by smithy-go-codegen DO NOT EDIT.`

## Generated vs hand-written
- **Generated:** `service/<svc>/api_*.go`, `types/`, `serializers.go`,
  `deserializers.go`, `validators.go`.
- **Hand-written, in the same package, ordinary filenames:** e.g. S3's
  [`handwritten_paginators.go`](https://github.com/aws/aws-sdk-go-v2/blob/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/service/s3/handwritten_paginators.go),
  `presign_post.go`, `express*.go`. The middleware stack (signing/retry/etc.)
  lives in `smithy-go` + `aws/` and is wired in by codegen.

## Customization
**Smithy integrations via Java SPI** (`GoIntegration`, registered in
[`META-INF/services/...GoIntegration`](https://github.com/aws/aws-sdk-go-v2/blob/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/codegen/smithy-aws-go-codegen/src/main/resources/META-INF/services/software.amazon.smithy.go.codegen.integration.GoIntegration)),
with a large per-service customization set under
[`customization/service/{s3,sqs,...}`](https://github.com/aws/aws-sdk-go-v2/tree/13bee3c72e77d68d6180fb633a7ce1c8714d53c0/codegen/smithy-aws-go-codegen/src/main/java/software/amazon/smithy/aws/go/codegen/customization/service)
(e.g. `SQSValidateMessageChecksum`, `LambdaRecursionDetection`, the ~20 S3
customizations).

## Multi-language relationship
A per-language Smithy generator; shares the Smithy core/trait model with the
TS/Rust/Python generators. **Monorepo, per-service Go modules** (each
`service/<svc>` has its own `go.mod`/`CHANGELOG.md`), versioned independently and
coordinated via `modman.toml`.

## Cross-language verification
Generated `internal/protocoltest/*` Go tests are compiled from the same Smithy
protocol-test traits as the JS suite (matching case IDs); endpoint tests
(`endpoints_test.go`, `TestEndpointCaseN`) follow the same pattern — see the AWS
verification section of `../CODE_GENERATION.md`.
