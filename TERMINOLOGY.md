## Core LLM Terms

### LLM (Large Language Model)
A neural network trained on vast amounts of text data to understand and generate human language. Examples include GPT-4, Claude, Llama, and Gemini. LLMs predict the next token(s) given a sequence of input tokens.

**Key Characteristics**:
- Billions to trillions of parameters
- Trained on internet-scale text corpora
- Emergent abilities at scale (100B+ parameters)
- Foundation for AI agents

---

### Token
The basic unit of text processing in LLMs. Tokens can be words, parts of words, or characters.

**Examples**:
- "Chat" → 1 token
- "GPT" → 1 token  
- "unbelievable" → 3 tokens ("un", "believ", "able")
- "Hello world" → 3 tokens ("Hello", " world", "<|endoftext|>")

**Approximate Rule of Thumb**: ~0.75 words per token (or ~1.3 tokens per word).

---

### Context Window / Context Length
The maximum number of tokens an LLM can process in a single inference call (input + output combined).

| Model | Context Window |
|-------|---------------|
| GPT-3.5 | 16K tokens |
| GPT-4 | 128K tokens |
| Claude 3.5 Sonnet | 200K tokens |
| Claude 3 Opus | 200K tokens |
| Llama 3.1 | 128K tokens |
| Gemini 1.5 Pro | 2M tokens |

**Importance for Agents**: Larger windows enable more history, more retrieved documents, and longer reasoning chains.

---

### Prompt
The input text provided to an LLM to elicit a desired response.

**Types**:
- **User prompt**: Direct user query
- **System prompt**: Instructions setting behavior ("You are a helpful assistant")
- **Few-shot prompt**: Includes examples of desired input/output pairs

---

### Prompt Engineering
The practice of designing and optimizing prompts to get better outputs from LLMs without changing model parameters.

**Techniques**:
- Chain-of-Thought prompting
- Few-shot examples
- Role assignment
- Output format specification
- Constraint specification

**Evolution**: Prompt Engineering → Context Engineering → Agent Engineering

---

### Completion
The output text generated by an LLM in response to a prompt.

**Types**:
- **Streaming completion**: Tokens delivered as generated (real-time)
- **Non-streaming**: Entire response delivered at once

---

### Temperature
A sampling parameter controlling randomness in generation.

- **Range**: 0.0 to 2.0 (typically 0.0 - 1.0)
- **Low (0.0-0.3)**: Deterministic, focused, repetitive
- **Medium (0.4-0.7)**: Balanced creativity and coherence
- **High (0.8-1.0+)**: Creative, diverse, potentially chaotic

**Formula**: Softmax(logits / temperature)

**Use Cases**:
- Coding: 0.0-0.2 (precise)
- Creative writing: 0.7-1.0 (varied)
- Chat: 0.5-0.7 (balanced)

---

### Top-p (Nucleus Sampling)
A sampling method that selects from the smallest set of tokens whose cumulative probability exceeds p.

- **Range**: 0.0 to 1.0
- **0.1**: Very conservative (top 10% probability mass)
- **0.9**: More diverse (top 90% probability mass)
- **1.0**: Consider all tokens

**How it works**: 
1. Sort tokens by probability
2. Accumulate until sum ≥ p
3. Sample from this "nucleus"

**Benefit**: Dynamically adjusts candidate set size based on confidence.

---

### Top-k
Sampling restricted to the k most likely tokens.

- **k=1**: Greedy decoding (always pick most likely)
- **k=10**: Sample from top 10
- **k=50**: Sample from top 50

**Comparison with Top-p**:
- Top-k: Fixed number of candidates
- Top-p: Dynamic number based on probability distribution

**Common Settings**: Top-p=0.9, Top-k=40, Temperature=0.7

---

### Hallucination
When an LLM generates plausible-sounding but false or ungrounded information.

**Types**:
- **Factual hallucination**: Incorrect facts ("The capital of France is London")
- **Faithfulness hallucination**: Output contradicts input/source
- **Intrinsic**: Contradicts training knowledge
- **Extrinsic**: Can't be verified from training data

**Mitigation**: RAG, grounding, verification, constrained generation

---

### Perplexity
A measure of how well a model predicts a sample. Lower is better.

**Formula**: exp(-average log probability of tokens)

**Interpretation**:
- Perplexity = 100: Model as confused as choosing from 100 options
- Perplexity = 10: Model confident (choosing from 10 options)

**Use**: Evaluating models, not for end users.

---

### Emergent Abilities
Capabilities that appear in LLMs only above a certain scale (parameter count).

**Examples**:
- In-context learning (emerges ~100B parameters)
- Chain-of-thought reasoning
- Instruction following
- Few-shot learning

**Implication**: Larger models can do qualitatively different things, not just quantitatively better.

---

### In-Context Learning (ICL)
The ability of LLMs to learn tasks from examples provided in the prompt, without parameter updates.

**Types**:
- **Zero-shot**: No examples, just instructions
- **One-shot**: One example
- **Few-shot**: Multiple examples (typically 2-8)

**Mechanism**: Not fully understood; likely pattern matching and induction heads.

---

### Instruction Tuning
Fine-tuning LLMs on datasets of (instruction, response) pairs to make them better at following user instructions.

**Process**:
1. Start with base (pre-trained) model
2. Create instruction-following dataset
3. Fine-tune model on instructions
4. Results in "chat" or "instruct" model

**Examples**: GPT-3 → GPT-3.5, Llama → Llama Chat

## AI Agent Terms

### AI Agent
An autonomous system that uses an LLM as its "brain" to perceive environment, make decisions, and take actions to achieve goals.

**Components**:
- **LLM Core**: Reasoning and decision-making
- **Tools**: Actions in environment
- **Memory**: State persistence
- **Planning**: Goal decomposition

**Key Capability**: Can act autonomously over multiple steps.

---

### Agent Loop / Cognitive Architecture
The basic cycle of agent operation:

```
┌─────────────┐
│   Observe   │ ◄── Get input from environment
└──────┬──────┘
       ▼
┌─────────────┐
│    Think    │ ◄── LLM reasons about observation
└──────┬──────┘
       ▼
┌─────────────┐
│    Act      │ ◄── Execute tool/action
└──────┬──────┘
       │
       └──────► Environment changes
                (loop back to Observe)
```

---

### Tool / Function
An external capability an agent can invoke.

**Examples**:
- Search the web
- Read a file
- Execute code
- Query a database
- Call an API

**Tool Use**: LLM generates structured output (function call) rather than free text.

---

### Tool Calling / Function Calling
The capability of LLMs to output structured function invocations instead of just text.

**Format**:
```json
{
  "name": "get_weather",
  "arguments": {
    "location": "San Francisco"
  }
}
```

**Flow**:
1. User asks question
2. LLM decides tool is needed
3. LLM outputs function call
4. System executes function
5. Result returned to LLM
6. LLM generates final response

---

### Planning
The ability to break down complex goals into subtasks and create action sequences.

**Types**:
- **Single-step**: One action at a time
- **Multi-step**: Pre-planned sequence
- **Hierarchical**: High-level plan → low-level actions
- **Reactive**: Replan based on observations

---

### Action
A specific operation an agent performs in the environment.

**Examples**:
- `file_read(path="/tmp/data.txt")`
- `web_search(query="current weather")`
- `code_execute(code="print(2+2)")`

---

### Observation
The result or feedback an agent receives after taking an action.

**Examples**:
- File contents after read
- Search results
- Command output
- Error messages

---

### State
The current situation of the agent including:
- Conversation history
- Memory contents
- Environment state
- Goal progress

---

### Autonomy
The degree to which an agent can operate without human intervention.

**Levels**:
- **Level 0**: Direct control (every action approved)
- **Level 1**: Suggest + approve
- **Level 2**: Act with constraints
- **Level 3**: Fully autonomous (Yolo mode)

---

### Multi-Agent System
A system with multiple AI agents that interact to solve problems.

**Patterns**:
- **Hierarchical**: Boss delegates to workers
- **Peer-to-peer**: Agents collaborate equally
- **Competitive**: Agents compete (best solution wins)
- **Pipeline**: Output of one feeds into next

**Examples**: MetaGPT, AutoGen, CrewAI

---

### Agent Framework
Software infrastructure for building AI agents.

**Popular Frameworks**:
- **LangChain**: Chains and agents
- **AutoGen**: Multi-agent conversations
- **LlamaIndex**: RAG-focused
- **CrewAI**: Role-based agents
- **OpenCode**: CLI-first agents

---

### Yolo Mode
Slang for fully autonomous mode where agent takes actions without human approval.

**Origin**: "You Only Live Once" → agent goes for it without asking.

**Risk**: Can make irreversible changes (delete files, send emails).

**Safety**: Usually requires explicit enablement.

---

### Human-in-the-Loop (HITL)
Design pattern where humans provide input or approval at key decision points.

**Use Cases**:
- Approving tool use
- Clarifying ambiguous requests
- Reviewing agent outputs
- Providing feedback

---

## Context & Memory

### Context
The information provided to an LLM for a specific inference call (prompt + any retrieved information).

**Components**:
- System instructions
- Conversation history
- Retrieved documents
- Tool descriptions
- Current task

---

### Context Engineering
The systematic optimization of information provided to LLMs, going beyond prompt engineering to include retrieval, processing, and management.

**Components**:
- Context retrieval (what to include)
- Context processing (how to structure)
- Context management (memory, caching)

---

### Context Window Management
Techniques for fitting relevant information within limited context window.

**Strategies**:
- **Compaction**: Summarize older content
- **Selection**: Choose most relevant
- **Prioritization**: Keep important, drop less important
- **Chunking**: Split across multiple calls

---

### Context Rot
The degradation of agent performance as conversation history grows and becomes less relevant or coherent.

**Symptoms**:
- Repetitive questions
- Contradictory actions
- Forgetting earlier decisions
- Wrong assumptions

**Mitigation**: Regular compaction, memory files, summarization.

---

### Memory
Persistent storage of information across sessions or long conversations.

**Types**:

#### Short-Term Memory
- Conversation history within context window
- Lost when context cleared

#### Long-Term Memory
- Persistent storage across sessions
- Types:
  - **Episodic**: Past interactions/experiences
  - **Semantic**: Facts and knowledge
  - **Procedural**: Skills and how-to knowledge

---

### Episodic Memory
Memory of specific events and experiences.

**Examples**:
- "User asked about Python last Tuesday"
- "Previous session worked on auth feature"
- "Agent tried approach X and it failed"

**Use**: Personalization, avoiding repeated mistakes, context resumption.

---

### Semantic Memory
General knowledge and facts independent of specific experiences.

**Examples**:
- "Python uses indentation for blocks"
- "The project uses React"
- "Best practice is to validate inputs"

**Use**: Domain knowledge, project context, coding standards.

---

### Procedural Memory
Knowledge of how to perform tasks (skills).

**Examples**:
- How to write a unit test
- How to debug a Python script
- How to structure a React component

**Use**: Consistent execution, skill reuse.

---

### Vector Database / Vector Store
A database optimized for storing and querying high-dimensional vectors (embeddings).

**Use Cases**:
- Semantic search
- RAG document retrieval
- Memory retrieval
- Similarity matching

**Popular Options**: Pinecone, Weaviate, Chroma, Qdrant, Milvus

---

### Embedding
A high-dimensional vector representation of text that captures semantic meaning.

**Properties**:
- Similar texts → similar vectors (cosine similarity)
- Dense representation (768-4096 dimensions typical)
- Computed by embedding models (e.g., text-embedding-3, BGE)

**Use**: Retrieval, clustering, classification, semantic search.

## Reasoning & Planning

### Chain-of-Thought (CoT)
Prompting technique where LLM generates intermediate reasoning steps before final answer.

**Format**:
```
Q: What is 15 * 24?
A: Let me break this down:
   15 * 20 = 300
   15 * 4 = 60
   300 + 60 = 360
   So 15 * 24 = 360
```

**Benefits**: Better reasoning, interpretability, error detection.

---

### ReAct (Reasoning + Acting)
Pattern combining reasoning traces with actions in an interleaved manner.

**Format**:
```
Thought: I need to find X
Action: search[X]
Observation: Results...

Thought: Now I can do Y
Action: calculate[...]
Observation: Result...

Thought: The answer is Z
Action: finish[Z]
```

**Benefits**: Grounded reasoning, error recovery, interpretability.

---

### Tree of Thoughts (ToT)
Extension of CoT that maintains multiple reasoning paths and explores them systematically.

**Process**:
1. Generate multiple thought candidates
2. Evaluate each
3. Select most promising
4. Continue or backtrack

**Use**: Complex problems requiring exploration (math, planning, games).

---

### Reflection / Reflexion
Pattern where agent reflects on past actions and outcomes to improve future performance.

**Process**:
```
Act → Observe result → Reflect on success/failure → 
Update strategy → Try again
```

**Benefits**: Self-improvement, learning from mistakes.

---

### Planning
The cognitive process of creating a sequence of actions to achieve a goal.

**Types**:
- **Classical planning**: Predefined steps
- **Hierarchical**: High-level plan → sub-plans
- **Adaptive**: Adjust plan based on feedback
- **LLM-based**: LLM generates plan

---

### Subgoal Decomposition
Breaking a complex goal into smaller, manageable subgoals.

**Example**:
```
Goal: "Build a website"
Subgoals:
1. Set up project structure
2. Create HTML layout
3. Add CSS styling
4. Implement JavaScript
5. Test and deploy
```

---

### Self-Consistency
Generating multiple reasoning paths and taking the majority vote.

**Process**:
1. Sample multiple CoT outputs
2. Extract answers
3. Take most common answer

**Benefits**: Improves accuracy, reduces random errors.

## Tool Use & MCP

### MCP (Model Context Protocol)
An open standard by Anthropic for connecting AI assistants to external data sources and tools.

**Purpose**: Standardized, secure, two-way connections between AI models and external systems.

**Architecture**: Client-server model using JSON-RPC 2.0

---

### MCP Server
A server that exposes tools, resources, and prompts via the MCP protocol.

**Capabilities**:
- **Tools**: Functions models can call
- **Resources**: Data models can read
- **Prompts**: Pre-defined templates

**Examples**: Filesystem server, GitHub server, PostgreSQL server

---

### MCP Client
An application that connects to MCP servers and uses their capabilities.

**Examples**: Claude Desktop, OpenCode, Claude Code

---

### Skill
A reusable capability or expertise module for an agent.

**Characteristics**:
- Pre-defined instructions
- Specific domain knowledge
- Triggered by context
- Examples: Code review skill, Testing skill, Documentation skill

**vs Tool**: Skill = knowledge + patterns; Tool = executable action

---

### Resource (MCP)
A piece of data exposed by an MCP server that models can read.

**Examples**:
- File contents
- Database records
- API responses
- System information

---

### JSON-RPC
A remote procedure call protocol using JSON for encoding. MCP uses JSON-RPC 2.0.

**Format**:
```json
{
  "jsonrpc": "2.0",
  "method": "tool/call",
  "params": {...},
  "id": 1
}
```

## RAG & Retrieval

### RAG (Retrieval-Augmented Generation)
Architecture that enhances LLM generation by retrieving relevant documents and including them in context.

**Flow**:
```
Query → Retrieve documents → Combine with query → 
Generate answer using both
```

**Benefits**: Reduced hallucination, up-to-date info, source attribution.

---

### Retrieval
The process of finding relevant documents given a query.

**Methods**:
- **Dense**: Vector similarity (embedding-based)
- **Sparse**: Keyword matching (BM25, TF-IDF)
- **Hybrid**: Combination of both

---

### Chunking
Dividing documents into smaller pieces for retrieval.

**Strategies**:
- Fixed size (e.g., 500 tokens)
- Semantic (by paragraph/section)
- Recursive (hierarchical)
- Sliding window (with overlap)

---

### Re-ranking
Applying a more expensive but accurate model to re-score retrieved documents.

**Flow**:
```
Initial retrieval (fast, approximate) → Top 100
→ Re-ranking (slow, accurate) → Top 5
```

---

### BM25
A sparse retrieval algorithm based on term frequency and inverse document frequency.

**Use**: Keyword-based search, good for exact matches.

---

### Cosine Similarity
A measure of similarity between two vectors based on the cosine of the angle between them.

**Range**: -1 to 1 (1 = identical direction)

**Use**: Finding similar embeddings.

---

### ANN (Approximate Nearest Neighbor)
Algorithms for efficiently finding similar vectors in high-dimensional space without exhaustive search.

**Examples**: HNSW, IVF, LSH

**Trade-off**: Speed vs accuracy (approximate vs exact).

## Training & Fine-tuning

### Pre-training
Training an LLM on a large corpus to learn general language understanding.

**Data**: Internet text, books, code
**Objective**: Next token prediction
**Cost**: Millions of dollars
**Result**: Base model (e.g., GPT-3, Llama-3)

---

### Fine-tuning
Adapting a pre-trained model to specific tasks or domains by training on task-specific data.

**Types**:
- **Full fine-tuning**: Update all parameters
- **PEFT**: Parameter-efficient (LoRA, adapters)
- **SFT**: Supervised fine-tuning on labeled data

---

### LoRA (Low-Rank Adaptation)
PEFT method that adds small trainable matrices to attention layers instead of updating all parameters.

**Benefits**:
- 99%+ parameter reduction
- Faster training
- Smaller model checkpoints
- Can swap adapters at runtime

---

### QLoRA
Quantized LoRA that loads base model in 4-bit precision while training adapters in full precision.

**Benefits**: Train large models on consumer GPUs.

---

### RLHF (Reinforcement Learning from Human Feedback)
Training method using human preferences to align models with human values.

**Process**:
1. Train reward model on human preferences
2. Fine-tune LLM using PPO to maximize reward
3. Result: Helpful, harmless, honest model

---

### PPO (Proximal Policy Optimization)
Reinforcement learning algorithm used in RLHF to optimize the language model.

**Characteristic**: Constrains how far policy can change to maintain stability.

---

### DPO (Direct Preference Optimization)
Alternative to RLHF that directly optimizes on preference data without explicit reward model.

**Benefits**: Simpler, more stable than PPO.

---

### Quantization
Reducing numerical precision of model weights to decrease memory and compute requirements.

**Formats**:
- **FP32**: Full precision (32-bit)
- **FP16**: Half precision
- **BF16**: Brain float (better range than FP16)
- **INT8**: 8-bit integer
- **INT4**: 4-bit integer (GGUF, AWQ)

**Trade-off**: Smaller/faster vs accuracy.

---

### Gradient Checkpointing
Memory optimization technique trading compute for memory by recomputing activations during backward pass.

**Benefit**: Train larger models with limited GPU memory.

---

## Inference & Performance

### Inference
The process of generating outputs from a trained model.

**Types**:
- **Prefill**: Process input tokens (compute-intensive)
- **Decode**: Generate output tokens (memory-bandwidth-bound)

---

### Latency
Time delay between request and response.

**Types**:
- **TTFT (Time To First Token)**: Until first token generated
- **TBT (Time Between Tokens)**: Inter-token latency
- **Total latency**: End-to-end time

**Units**: Milliseconds (ms), seconds (s)

---

### Throughput
Rate at which tokens are generated (tokens per second).

**Measurement**: Output tokens / total time

**Optimization**: Batching, quantization, efficient kernels.

---

### KV Cache
Key-value cache storing attention computations from previous tokens to avoid recomputation.

**Benefit**: Massive speedup for autoregressive generation.

**Cost**: Memory usage grows with sequence length.

---

### Batching
Processing multiple requests together to improve GPU utilization.

**Types**:
- **Static**: Fixed batch size
- **Dynamic**: Varying batch size
- **Continuous**: Add/remove requests dynamically

---

### Speculative Decoding
Speedup technique using a smaller draft model to predict tokens, verified by main model.

**Speedup**: 2-3x when draft model is accurate.

---

### Flash Attention
Memory-efficient attention algorithm that reduces HBM accesses.

**Benefits**: Faster, memory-efficient, enables longer sequences.

---

### vLLM
High-throughput LLM serving engine using PagedAttention for efficient KV cache management.

**Features**: Continuous batching, PagedAttention, tensor parallelism.

---

## Evaluation & Benchmarks

### Benchmark
Standardized test for evaluating model capabilities.

**Types**:
- **Knowledge**: MMLU, TriviaQA
- **Reasoning**: GSM8K, HumanEval, MATH
- **Agents**: SWE-bench, WebArena, OSWorld
- **Chat**: MT-bench, AlpacaEval

---

### Pass@k
Metric measuring if any of k generated samples passes tests.

**Use**: Code generation evaluation.

**Example**: Pass@1 = 50%, Pass@10 = 80%

---

### BLEU / ROUGE
Metrics for evaluating text generation quality by comparing to reference.

**BLEU**: Precision-focused (translation)
**ROUGE**: Recall-focused (summarization)

---

### MMLU (Massive Multitask Language Understanding)
Benchmark testing knowledge across 57 subjects (STEM, humanities, social sciences).

**Scores**:
- Random: 25%
- GPT-3.5: ~70%
- GPT-4: ~86%
- Human expert: ~89%

---

### HumanEval
Benchmark of 164 hand-written programming problems.

**Metric**: Pass@k (functional correctness)

---

### SWE-bench
Benchmark of real GitHub issues requiring models to generate patches.

**Difficulty**: Requires repository understanding, debugging, implementation.

---

### Time Horizon
The length of task (in human time) that an AI agent can complete with given reliability.

**Trend**: Doubling every ~7 months (METR research).

---

## Architecture & Models

### Transformer
The neural architecture underlying modern LLMs, based on attention mechanisms.

**Key Components**:
- **Self-attention**: Weighted aggregation of tokens
- **Feed-forward**: Point-wise transformation
- **Layer normalization**: Stabilize training

**Paper**: "Attention Is All You Need" (2017)

---

### Attention Mechanism
Mechanism allowing model to focus on relevant parts of input when generating output.

**Types**:
- **Self-attention**: Attend to own tokens
- **Cross-attention**: Attend to other sequence
- **Multi-head**: Multiple attention heads in parallel

**Complexity**: O(n²) with sequence length.

---

### Feed-Forward Network (FFN)
The MLP component of transformer layers applied to each position independently.

**Structure**: Linear → Activation → Linear

---

### Layer Normalization
Technique normalizing inputs across feature dimension to stabilize training.

**Variants**: Pre-norm (more stable), Post-norm (original)

---

### Positional Encoding
Method to inject position information into token embeddings (since attention is permutation-invariant).

**Types**:
- **Sinusoidal**: Fixed periodic functions
- **Learned**: Trainable position embeddings
- **RoPE**: Rotary position embeddings (modern)
- **ALiBi**: Attention with linear biases

---

### Tokenizer
Component that converts text to tokens and back.

**Types**:
- **BPE**: Byte Pair Encoding (GPT, Llama)
- **WordPiece**: (BERT)
- **SentencePiece**: (T5, Llama)
- **Tiktoken**: (OpenAI)

**Vocabulary Size**: 32K-200K tokens typical.

---

### Base Model / Foundation Model
A pre-trained model before any task-specific fine-tuning.

**Examples**: GPT-3, Llama-3-base, Claude-base

---

### Instruct Model / Chat Model
A fine-tuned model optimized for following instructions and conversation.

**Examples**: GPT-3.5-turbo, Llama-3-instruct, Claude-3

---

### MoE (Mixture of Experts)
Architecture using multiple "expert" networks with a gating mechanism.

**Benefit**: Scale parameters without proportional compute increase.

**Examples**: GPT-4 (rumored), Mixtral 8x7B

---

### Causal/Masked Language Modeling
Training objective where model predicts next token (causal) or masked tokens (BERT-style).

**Causal**: Autoregressive (GPT-style)
**Masked**: Bidirectional (BERT-style)
